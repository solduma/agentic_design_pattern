# AI 개발의 경제학

AI가 소프트웨어 개발 수명주기에 미치는 영향을 평가할 때, 대화는 대부분 개발자 속도—코드를 얼마나 빨리 작성할 수 있는가—에서 시작해 그것으로 끝납니다. 그러나 엔지니어링 리더에게는 더 중요한 지표인 총소유비용(TCO, Total Cost of Ownership)이 있습니다. AI 보조 개발의 진정한 비용을 이해하려면, 서로 다른 워크플로우가 자본적 지출(CapEx, Capital Expenditure)—무언가를 구축하기 위한 선행 투자—과 운영 지출(OpEx, Operational Expenditure)—구축한 것을 운영·수정·유지보수하는 지속 비용—사이에서 재무·운영 부담을 어떻게 이동시키는지 살펴봐야 합니다. 결정적으로, AI 시대에는 OpEx가 토큰 경제(token economy)에 크게 좌우됩니다.

## 바이브 코딩의 숨겨진 부채 (저CapEx, 고OpEx)

얼핏 보면 바이브 코딩은 놀랍도록 비용 효율적으로 보입니다. 진입 장벽이 사실상 없습니다. AI 어시스턴트의 표준 월정액 구독과 몇 번의 가벼운 프롬프트만 있으면 됩니다. 개발자가 시스템 설계에 시간을 투자하는 대신 모델의 기본 역량에 전적으로 의존하기 때문에 CapEx는 미미합니다.

그러나 바이브 코딩의 경제학에는 막대하고 복리로 불어나는 OpEx 부담이 숨어 있습니다.

- **토큰 소모율:** 대규모 언어 모델(Large Language Model, LLM)과의 모든 상호작용에는 입력·출력 토큰에 따른 비용이 발생합니다. 바이브 코딩에서 개발자는 거대하고 구조화되지 않은 파일을 컨텍스트 윈도우에 통째로 쏟아 넣고, 모델이 스스로 검증하지 않은 실수를 반복 수정하도록 요청하는 경우가 많습니다. 이는 1회 성공률이 낮은 채 API 토큰을 소진하는 값비싼 "프롬프팅 루프"를 만들어 냅니다.

- **유지보수 부담:** 즉흥적인 프롬프팅으로 작성된 코드는 구조적 일관성이 부족한 경우가 많습니다. 6개월 후 버그가 발생하면 인간 엔지니어가 구조화되지 않은 AI 생성 "스파게티" 코드를 역공학하는 데 며칠을 소비해야 합니다.

- **보안 수정 비용:** 자동화된 평가 하네스 없이 코드를 빠르게 생성하면 취약점도 빠르게 생성됩니다. 프로덕션에서 보안 결함을 수정하는 비용은 설계 단계에서 발견했을 때보다 기하급수적으로 높습니다.

## 에이전틱 엔지니어링의 투자 (고CapEx, 저OpEx)

에이전틱 엔지니어링은 이 경제 모델을 뒤집습니다. 프로덕션 코드 한 줄이 생성되기 전에 엔지니어링 시간과 리소스를 의도적으로 선행 투자해야 합니다.

에이전틱 엔지니어링의 CapEx에는 API 스키마 설계, 결정론적 테스트 스위트 구축, 그리고 가장 중요하게는 에이전트의 컨텍스트 구조화가 포함됩니다. 이 선행 비용이 더 높지만, 기능을 배포하고 유지보수하는 한계 비용은 극적으로 낮아집니다. AI가 엄격하게 관리되는 "공장" 안에서 동작하기 때문에 그 출력은 구조적으로 견고하고 사전 테스트되어 있으며 회사 표준에 부합합니다.

## 재무 레버로서의 컨텍스트 엔지니어링

토큰 경제에서 컨텍스트 엔지니어링은 단순한 기술 역량이 아니라 재무 전략입니다. LLM은 전송하는 모든 정보에 비용을 청구합니다. 100,000토큰짜리 저장소 전체를 모든 프롬프트에 넣는 것은 규모가 커지면 재무적으로 지속 불가능합니다.

효과적인 컨텍스트 엔지니어링은 모델이 광범위하고 노이즈가 많은 입력 데이터(payload) 대신, 정밀한 AGENTS.md 파일과 아키텍처 가드레일 같은 밀도 높고 신호가 풍부한 입력 데이터를 수신하도록 보장합니다. 적절한 컨텍스트를 미리 제공함으로써 개발자는 에이전트의 1회 성공률을 대폭 높이고, 바이브 코딩을 괴롭히는 값비싼 시행착오 루프를 피할 수 있습니다.

## 동적 컨텍스트와 스킬을 통한 효율 확장

OpEx를 진정으로 최적화하려면 고급 에이전틱 엔지니어링은 모델 컨텍스트 프로토콜(MCP) 서버 같은 "스킬" 또는 도구 호출을 통한 동적 컨텍스트에 의존합니다. 이 주제는 3일차 백서에서 자세히 다룹니다.

## 지능형 모델 라우팅

나아가 에이전틱 엔지니어링은 지능형 모델 라우팅(intelligent model routing)을 가능하게 합니다. 바이브 코딩 워크플로우에서 개발자는 일반적으로 모든 상호작용에 단일 대형 최첨단 모델(frontier model)에만 의존합니다. AI에게 오타 수정이나 기본 단위 테스트 생성을 요청할 때조차 프리미엄 토큰 가격을 지불하는 것입니다.

잘 설계된 공장 모델은 이러한 낭비를 피합니다. 고도로 복잡한 작업(요구사항 정의, 아키텍처, 초기 구현)에는 크고 고급 모델을 사용하지만, 결정론적이고 저복잡도인 작업(테스트 생성, 코드 리뷰, CI/CD 모니터링)은 더 작고 빠르며 훨씬 저렴한 모델로 자동 라우팅합니다. 다중 모델 생태계를 오케스트레이션함으로써 엔지니어링 팀은 최고 수준의 출력 품질을 유지하면서 운영 토큰 비용을 체계적으로 낮출 수 있습니다.
