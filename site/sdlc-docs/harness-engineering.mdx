# 하네스 엔지니어링: 모델을 둘러싼 것

개발자들이 AI 에이전트와 일하기 시작할 때, 모델 자체가 곧 에이전트 시스템이라 여기기 쉽습니다. 새 모델이 출시되면 에이전트가 더 똑똑해지고, 오래된 모델을 쓰면 에이전트가 나빠진다는 것입니다. 모델이 좋고 나쁜 모든 결과의 원인으로 지목됩니다.

그 직관은 틀렸으며, 잘못된 투자로 이어집니다. 모델은 실행 중인 에이전트를 구성하는 요소 중 하나입니다. 나머지 모든 것 — 프롬프트, 도구, 컨텍스트 정책, 훅(hooks), 샌드박스(sandbox), 서브에이전트, 관측성(observability) — 이 바로 하네스(harness)입니다. 모델이 실제로 무언가를 완수하게 만드는 발판(scaffolding)입니다.[^11]

날것의 모델은 에이전트가 아닙니다. 하네스가 상태, 도구 실행, 피드백 루프, 강제할 수 있는 제약을 갖출 때 비로소 에이전트가 됩니다. 개발자가 Claude Code, Cursor, Codex, Antigravity, Aider 또는 Cline을 사용하면서 경험하는 동작은, 어떤 모델이 내부에서 돌아가느냐가 아니라 하네스가 무엇을 하느냐에 따라 좌우됩니다.

## 하네스의 구성 요소

구체적으로, 하네스는 다음을 포함합니다:

- **지시 및 규칙 파일(Instructions and Rule Files)**: 에이전트가 누구이고, 무엇을 중시하며, 무엇이 금지되어 있는지를 정의하는 텍스트. AGENTS.md, CLAUDE.md, GEMINI.md, 스킬 파일, 서브에이전트 프롬프트가 이에 해당합니다.
- **도구(Tools)**: 에이전트가 호출할 수 있는 함수, MCP 서버, API. 그리고 모델에게 언제 어떻게 호출할지를 알려 주는 설명 문구도 포함됩니다.
- **샌드박스와 실행 환경**: 에이전트의 코드가 실제로 실행되는 곳, 접근할 수 있는 것과 접근할 수 없는 것.
- **오케스트레이션 로직(orchestration logic)**: 서브에이전트 생성, 모델 라우팅, 전문가 간 핸드오프, 그리고 각각의 발동 조건(trigger condition)을 결정하는 기준.
- **가드레일(guardrails) 또는 훅**: 특정 수명주기 지점 — 도구 호출 전, 파일 편집 후, 커밋 전 — 에서 실행되는 결정론적 코드. 에이전트가 절대 잊어서는 안 되지만 자주 잊는 것들을 위한 자리입니다.
- **관측성**: 로그, 트레이스, 평가, 비용 및 지연 계측(metering). 관측성이 없으면 에이전트가 잘하고 있는지 조용히 이탈(drift)하고 있는지 알 방법이 없습니다.

관리해야 할 영역이 많아 보인다면, 실제로 그렇습니다. 그리고 그 책임 영역은 팀의 것이지, 모델 공급자의 것이 아닙니다.

## SDLC에서의 하네스

모델 자체는 작업을 어떻게 수행할지를 결정하지만, 하네스는 실제 실행에 필요한 도구·샌드박스·오케스트레이션으로 연결해 주는 발판입니다. 따라서 이 하네스는 AI 에이전트가 작동하는 모든 단계에 반드시 존재해야 합니다.

하네스가 새로운 SDLC의 각 단계에 걸쳐 어떻게 능동적으로 작동하는지 설명합니다:

### 1. 요구사항·계획·아키텍처 (하네스 구성)

이 단계는 하네스가 설정되고 조정되는 곳입니다. AI가 프로덕션 코드를 작성하기 전에 개발자는 에이전트의 환경을 구성해야 합니다.

- **하네스 구성**: 하네스가 로드하여 모델에게 제공할 지시 및 규칙 파일(예: AGENTS.md 작성 및 아키텍처 제약 정의)을 준비합니다.
- **행동**: 개발자는 에이전트가 접근할 도구(특정 API나 데이터베이스 스키마 등)를 정의하고, 에이전트가 위반할 수 없는 기본 규칙을 설정합니다.

### 2. 구현 (하네스 실행)

활발한 코딩 중에 하네스는 AI 모델이 작업에 집중하고 안전하며 생산적으로 동작하도록 지탱하는 틀로서 작동합니다.

- **사용되는 하네스 구성 요소**: 샌드박스, 실행 환경, 도구.
- **행동**: 모델이 코드를 생성하면 하네스의 격리된 샌드박스 내에서 실행됩니다. 모델이 파일을 읽거나 웹을 검색해야 할 때는 하네스가 제공하는 도구를 사용합니다.

### 3. 테스트 및 QA (피드백 루프)

에이전틱 워크플로우에서의 테스트는 자율적 자기 교정을 가능하게 하기 위해 하네스에 크게 의존합니다.

- **사용되는 하네스 구성 요소**: 오케스트레이션 로직, 가드레일.
- **행동**: 에이전트가 함수를 작성하면 하네스는 자동화된 테스트가 실행될 수 있도록 실행 환경(샌드박스 터미널 등)을 제공합니다. 테스트가 실패하면 오케스트레이션 로직은 해당 환경에서 오류 출력을 포착하여 모델에 다시 전달하고 재시도를 요청합니다. 하네스가 바로 이 자동화된 `think -> act -> observe` 루프를 만들어 냅니다.

### 4. 코드 리뷰·배포·유지보수 (하네스 관찰)

코드가 작성된 후에도 하네스는 라이브 또는 라이브에 가까운 환경에서 에이전트가 안전하게 동작하도록 보장합니다.

- **사용되는 하네스 구성 요소**: 훅, 관측성.
- **행동**: 하네스는 결정론적 훅을 실행합니다(예: 에이전트가 하드코딩된 비밀번호를 푸시하려 할 때 커밋 차단). 또한 관측성 레이어는 토큰 비용, 지연, 에이전트 이탈을 추적하여 인간 엔지니어가 에이전트의 특정 배포 결정 근거를 정확히 감사할 수 있게 합니다.

바이브 코딩에서 에이전틱 엔지니어링으로의 전환은 단순히 어떤 도구를 사용하느냐의 문제가 아닙니다. 개발자는 완전히 동일한 에이전트로 바이브 코딩을 할 수도, 에이전틱 엔지니어링을 적용할 수도 있습니다. 그 차이는 하네스를 얼마나 의도적으로 구성하고 적용하느냐에 달려 있습니다. 바이브 코딩은 빠른 구현만을 목표로 하는 최소화되거나 암묵적인 발판에 의존합니다. 에이전틱 엔지니어링은 AI를 초기 계획 문서 작성 단계부터 프로덕션 모니터링까지 이끄는 명확하고 포괄적인 하네스 설계에 의존합니다.

이 의도적인 구성의 효과는 수치로 분명히 확인됩니다. 공개 벤치마크들은 하네스 효과의 규모를 구체적으로 보여 줍니다. Terminal Bench 2.0에서 한 팀은 하네스만 변경하고 모델은 전혀 바꾸지 않은 채로 코딩 에이전트를 Top 30 밖에서 Top 5로 끌어올렸습니다. LangChain의 별도 연구에서는 고정된 모델 주변의 시스템 프롬프트, 도구, 미들웨어만 조정하여 동일 벤치마크에서 코딩 에이전트 점수를 13.7점 높였습니다.

이 교훈의 실전적 의미는 SDLC 전반에 AI를 도입하는 팀에게 결정적으로 중요합니다. 에이전트가 잘못된 행동을 하면 첫 번째 본능은 모델을 탓하는 것입니다. 그러나 실패 원인을 냉정히 살펴보면 대부분 누락된 도구, 모호한 규칙, 미비한 가드레일, 또는 노이즈로 가득 찬 컨텍스트 윈도우에 있습니다. 대부분의 에이전트 장애는 구성 실패입니다.
