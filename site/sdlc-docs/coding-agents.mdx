# 실무에서의 코딩 에이전트

오늘날 에이전트를 구축하는 개발자는 대부분의 작업을 터미널에서, 자연어로, 그리고 또 다른 코딩 에이전트가 타이핑을 대신하는 방식으로 처리합니다. 이것은 새로운 현상입니다. 1년 전만 해도 같은 작업은 프레임워크, SDK, 클라우드 콘솔을 의미했습니다. 이를 대체한 패턴들은 명확히 이름 붙일 가치가 있습니다. 하루 일과에 코딩 에이전트를 활용하고 싶은 개발자에게도, 자신만의 에이전트를 구축하고 싶은 개발자에게도 마찬가지입니다.

## 개발자의 일과에서 코딩 에이전트의 위치

코딩 에이전트는 일상적인 업무의 세 곳에서 등장합니다. 대부분의 개발자는 세 가지 모두를 동시에 사용합니다.

**에디터에서:** 개발자가 타이핑하는 동안 다음 줄을 제안하는 인라인 자동완성. 코드를 제자리에서 설명하거나 수정하는 채팅 패널. IDE 내부의 전체 코드베이스 인식. 대부분의 사람들이 코딩에서 처음 AI를 만나는 곳이며, 작업이 흐름(flow) 안에 머무는 곳입니다. 예시로는 GitHub Copilot, Cursor, Windsurf, JetBrains AI Assistant가 있습니다.

**터미널에서:** 개발자가 커맨드 라인에서 실행하고, 일상 언어로 목표를 전달하며, 코드베이스 전반에 걸쳐 작업하게 하는 코딩 에이전트. 전체 파일 시스템 접근, 멀티 파일 편집, 도구와 테스트를 실행하고 결과에 따라 반복하는 능력. 오늘날 진지한 바이브 코딩이 이루어지는 곳입니다. 예시로는 Antigravity CLI, Claude Code, Codex CLI, Open Code, Cline이 있습니다.

**백그라운드에서:** 작업을 받아 클라우드 호스팅 샌드박스에서 자율적으로 실행하는 에이전트. 종종 수 시간 동안 실행되며 풀 리퀘스트를 결과물로 생성합니다. 개발자는 작업을 넘기고 나중에 검토합니다. 예시로는 Google Jules, GitHub Copilot 에이전트 모드, Cursor의 백그라운드 에이전트, 그리고 고급 알고리즘 설계를 위한 Google의 특화 에이전트 AlphaEvolve가 있습니다.

실무에서, 에디터 에이전트는 개발자가 코드를 작성하는 중간에 흐름을 벗어나지 않고 제안·빠른 편집·설명을 원할 때 유용합니다. 터미널 에이전트는 멀티 파일 작업, 낯선 코드베이스 탐색, 에이전트가 코드를 실행하고 관찰한 결과에 반응해야 하는 작업에 적합합니다. 백그라운드 에이전트는 알려진 버그 수정, 테스트 스위트 생성, 한 프레임워크에서 다른 프레임워크로의 코드 마이그레이션처럼 한 문단으로 기술하고 자리를 비울 수 있는 잘 정의된 작업에 잘 맞습니다. 같은 개발자가 하루에 세 가지 모두를 사용하는 경우가 많습니다.

올바른 출발점은 어떤 카테고리가 자율성 사다리에서 더 높은 위치에 있는가가 아니라, 작업의 성격에 달려 있습니다.

## 바이브 코딩으로 프로덕션 수준 에이전트 구축하기

지금까지 논의한 모든 내용은 소프트웨어 구축에 코딩 에이전트를 사용하는 것이었습니다. 기능 작성, 버그 수정, 테스트 생성, 코드 리팩터링이 그것입니다. 하지만 구축해야 할 대상 자체가 에이전트라면 어떻게 될까요?

환불 요청을 처리하는 고객 지원 봇. 출처를 상호 참조하고 근거 있는 보고서를 생성하는 리서치 어시스턴트. 규정 준수를 모니터링하고 이상 징후를 신고하는 내부 도구. 이것들은 터미널의 코딩 에이전트로 해결하는 작업이 아닙니다. 자체 도구, 자체 메모리, 자체 평가, 자체 배포 인프라가 필요한 제품들입니다.

프로토타입 스크립트를 생성하던 동일한 터미널 기반 워크플로우가 이제 이러한 프로덕션 에이전트에도 도달합니다. 영구 메모리, 거버넌스, 관측성을 갖추고 규모에 맞게 실행되는 진짜 에이전트를 구축·평가·배포하는 작업이, 프레임워크와 클라우드 콘솔 과제에서 벗어나 개발자가 이미 사용하던 동일한 코딩 에이전트와 대화하는 방식으로 같은 터미널 안에서 이루어집니다.

이 워크플로우가 중요한 이유는 빌더가 실제 사용자를 위해 안정적으로 실행되는 에이전트를 필요로 할 때입니다. 세션 간 영구 메모리, 도구와 데이터에 대한 스코핑된 권한, 배포 전 회귀를 잡아내는 평가 커버리지, 에이전트가 실제로 한 일을 추적하는 관측성. 일회성 스크립트나 개인 자동화에는 일반 코딩 에이전트로 충분하며, 에이전트가 곧 목적지입니다. 실제 사용자에게 규모로 서비스하는 에이전트의 경우, 에이전트가 제품이며 그 아래의 기반이 필요합니다.

Google의 Agents CLI는 이 아이디어를 중심으로 구축되었습니다.[^14] 이 도구는 Google Cloud 위에서 에이전트를 구축하기 위한 스킬 세트를 묶은 소형 커맨드라인 도구로, 결정적으로 개발자가 선호하는 코딩 에이전트(Claude Code, Codex 또는 다른 것)와 함께 작동합니다. 일회성 설치 후, 코딩 에이전트는 에이전트 개발 키트(Agent Development Kit, ADK) 수명주기 전체를 아우르는 일곱 가지 새로운 스킬을 얻습니다. 프로젝트 스캐폴딩, 에이전트 코드 작성, 평가, Agent Runtime 배포, 관측성 연결이 그것입니다. 개발자는 새로운 SDK를 배울 필요가 없습니다. 원하는 바를 기술하면 코딩 에이전트가 각 단계에서 스킬을 사용해 올바른 일을 처리합니다.

구체적으로, 전체 구축-평가-배포 루프는 다음과 같습니다.

**Snippet 1: Agents CLI Setup and Build.**

```bash
# One-time setup
uvx google-agents-cli setup
# Then in your coding agent:
> Build a support agent that answers questions from our docs.
> evaluate it on the FAQ dataset 
> Deploy it to Agent Engine
```

단 하나의 지시 뒤에서, 코딩 에이전트는 템플릿에서 프로젝트를 스캐폴딩하고, ADK 코드를 작성하고, 평가셋을 생성하고, 에이전트에 대해 실행하고, Agent Runtime에 배포하고, 결과를 보고합니다. 직접 구동하는 것을 선호하는 개발자를 위해, 동일한 작업이 일반 CLI 명령(`agents-cli create`, `agents-cli playground`, `agents-cli eval`, `agents-cli deploy`)으로도 제공됩니다.

프로덕션 에이전트는 한때 프로토타입과 별개의 스택과 별개의 워크플로우가 필요했습니다. 이제 어제 개발자의 노트북에서 실행되던 프로토타입이 오늘 실제 사용자에게 서비스하는 프로덕션 에이전트가 될 수 있습니다. 재작성 없이.

동일한 워크플로우가 단일 에이전트에서 다수로 확장됩니다. ADK는 그래프 기반 워크플로우, 협업 에이전트 구축을 위한 멀티 에이전트 워크플로우, 그리고 공유 세션 상태, LLM 기반 위임, 명시적 호출 같은 상호작용 메커니즘을 제공하며, 이것들이 문제에 맞는 멀티 에이전트 패턴으로 조합됩니다.

에이전트 간 조율은 단순한 경우 공유 세션 상태를 통해, 도구 접근을 위해서는 모델 컨텍스트 프로토콜(Model Context Protocol, MCP)을 통해, 에이전트 간 위임을 위해서는 에이전트 간 프로토콜(Agent2Agent, A2A)을 통해 이루어집니다.[^15] Anthropic 엔지니어링 팀은 2026년 초에 이런 종류의 아키텍처 위에서 실행된 에이전트 팀이 2주에 걸쳐 동작하는 C 컴파일러를 Rust로 구축한 실험을 발표했습니다. 인간은 방향을 설정하고 결과물을 검토했지만 구현을 작성하지는 않았습니다.[^16] 병목은 코드를 작성하는 것에서 무엇을 해야 하는지 명세하고 에이전트가 그것을 수행했는지 검증하는 것으로 이동했습니다.

빌더에게 실질적인 시사점은 단순합니다. 오늘 스크립트를 생성하는 동일한 바이브 코딩 워크플로우가 내일 프로덕션 에이전트를 만듭니다. 수명주기(구축, 평가, 배포, 관측, 개선)가 한 곳에 삽니다. 아이디어에서 실행 중인 에이전트까지의 경로가 몇 주에서 몇 시간으로 단축되었으며, 대부분의 작업이 자연어로 이루어집니다.

이 워크플로우를 팀 규모에서 프로덕션 수준으로 만드는 실천들(스펙 주도 개발, 구조화된 코드 리뷰부터 가드레일, 샌드박싱, 제로 트러스트 개발까지)은 Day 5 동반 논문인 *Spec-Driven Production Grade Development in the Age of Vibe Coding*에서 다룹니다.
