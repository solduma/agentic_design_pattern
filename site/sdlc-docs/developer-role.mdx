# 진화하는 개발자의 역할: 지휘자와 오케스트레이터

AI가 구현 업무를 점점 더 많이 맡으면서, 개발자의 역할은 흥미롭고도 낯선 방식으로 변모하고 있습니다. 두 가지 모드를 생각해 보면 유용합니다. 개발자는 이 두 모드 사이를 유동적으로 오갑니다: 지휘자(conductor)와 오케스트레이터(orchestrator).[^12]

## 지휘자: 실시간 직접 지시

지휘자 모드에서는 개발자가 AI 페어 프로그래머와 실시간으로 협업합니다. IDE 안에서 코드가 나타나는 것을 지켜보고, 프롬프트와 수정으로 AI를 안내하며, 작성되는 내용에 대해 세밀한 통제권을 유지합니다. AI는 강력한 악기이지만, 개발자가 모든 움직임을 능동적으로 지휘합니다.

이 모드는 복잡한 로직 작업, 까다로운 이슈 디버깅, 또는 각 변경 사항을 즉시 파악해야 하는 낯선 코드베이스 작업에 전형적으로 나타납니다. GitHub Copilot, Google's Gemini Code Assist, Cursor, Windsurf 같은 도구들은 인라인 완성, 채팅 인터페이스, 즉석 편집 기능을 통해 이 모드를 주로 지원합니다.

지휘자 모드는 전통적인 엔지니어링 배경을 가진 개발자에게 자연스럽습니다. 많은 엔지니어가 중시하는 이해와 통제 감각을 보존하기 때문입니다. 리스크는 병목이 될 수 있다는 점입니다. 개발자가 모든 키 입력을 직접 지시한다면, AI로 인한 처리량 향상은 제한됩니다.

## 오케스트레이터: 비동기 멀티 에이전트 위임

오케스트레이터 모드에서는 개발자가 더 높은 추상화 수준에서 운영합니다. 목표를 정의하고 에이전트에 할당한 뒤 결과를 검토하지만, 코드가 한 줄씩 나타나는 것을 지켜보지는 않습니다. 에이전트들은 백그라운드에서, 병렬로, 코드베이스의 서로 다른 부분을 작업할 수 있습니다. 개발자는 주기적으로 점검하고, 출력을 검토하며, 방향을 교정합니다.

이 모드는 버그 수정, 확립된 패턴에 따른 기능 구현, 코드베이스 마이그레이션, 테스트 생성처럼 잘 정의된 작업에 전형적으로 나타납니다. Google's Jules, GitHub Copilot의 에이전트 모드, Cursor의 백그라운드 에이전트, Claude Code 같은 도구들은 비동기 작업 실행을 통해 이 모드를 지원하며, 종종 저장소·빌드 도구·테스트 스위트에 완전히 접근할 수 있는 샌드박스 환경에서 동작합니다.[^13]

오케스트레이터 모드는 다른 기술 집합을 요구합니다. 구문과 언어 관용구에 대한 깊은 전문 지식 대신, 다음과 같은 역량이 필요합니다:

- **명세(Specification)**: 에이전트가 모호함 없이 실행할 수 있을 만큼 작업을 정확하게 정의하는 능력
- **분해(Decomposition)**: 대규모 작업을 에이전트 실행에 적합한 크기의 단위로 쪼개는 능력
- **평가(Evaluation)**: 에이전트 출력이 품질 기준을 충족하는지 신속하게 판단하는 능력
- **시스템 설계(System design)**: 에이전트가 생산적으로 작동하도록 하는 제약·테스트·피드백 루프를 설계하는 능력

## 80% 문제

AI 지원 개발에서 지속적으로 제기되는 난제가 있습니다. 바로 80% 문제(the 80% problem)입니다. AI 에이전트는 기능 코드의 약 80%를 빠르게 생성할 수 있지만, 나머지 20%—엣지 케이스, 오류 처리, 통합 지점, 미묘한 정확성 요구사항—는 현재 모델이 종종 갖추지 못한 깊은 맥락 지식을 요구합니다.[^14]

AI 오류의 성격은 단순한 구문 실수에서 더 은밀한 개념적 실패로 진화했습니다. 비즈니스 로직에 대한 잘못된 가정, 모호한 요구사항에 대한 명확화 시도 부재, 누락된 엣지 케이스, 미묘한 장기 유지보수 부담을 야기하는 아키텍처 결정 등이 그 예입니다. 코드가 "올바르게 보이고" 기본 테스트조차 통과할 수 있기 때문에, 이러한 오류는 발견하기가 더 어렵습니다.

이 난제를 가장 효과적으로 헤쳐 나가는 개발자들은 특정한 자세를 채택합니다. AI가 잘하는 것(명확히 명세된 작업의 신속한 구현)에는 AI를 활용하되, AI가 어려움을 겪는 것(모호한 요구사항, 아키텍처적 트레이드오프, 정확성 검증)에는 자신의 주의를 집중시키는 것입니다. 그들은 AI가 생산한 모든 것을 수용함으로써 더 빨라지려 하지 않습니다. 자신의 전문성이 가장 중요한 곳에 집중함으로써 더 빨라지려 합니다.

80% 문제를 효과적으로 해결하려면 작업의 적절한 단계에 올바른 도구를 적용해야 합니다. '지휘자'로 운영하는 개발자는 '오케스트레이터'로 운영하는 개발자와는 다른 도구 집합이 필요합니다. 이 운영 모드를 일상적인 워크플로우에 어떻게 매핑할지 이해하려면, 자율성과 통합 방식을 기준으로 현재 AI 에이전트 지형을 분류해야 합니다.
