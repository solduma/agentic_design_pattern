"새 분석"에 평범한 한국어로 질문을 던지면, 마스터 라우터가 그 질문이 어느 분야인지 판별해 일곱 전문 에이전트 중 하나에게, 또는 어디에도 해당 안 되면 범용 에이전트에게 넘기거나, 여러 분야에 걸치면 사용자에게 되묻고(clarify) 선택을 기다립니다. 어느 경로든 동일한 공통 상태 머신(실행 루프)가 도구를 부르고 결과를 돌려줍니다.

1. 큰 그림

"새 분석" 화면에 질문을 입력하면 한 번의 요청이 POST /chat/ask로 들어옵니다. 그 뒤엔 네 개의 역할이 순서대로 맞물립니다.

비유

큰 병원의 접수 데스크를 떠올리세요. 환자(질문)가 오면 접수 담당(마스터 라우터)이 증상을 듣고 어느 전문의에게 보낼지 정합니다. 애매하면 "내과일까요, 외과일까요?"라고 되묻고(사람이 고르는 HITL(Human-in-the-Loop, 사람 개입)), 아예 맞는 과가 없을 때만 종합의(범용 에이전트)가 직접 봅니다. 어느 진료실이든 문진 → 처치 → 경과 관찰 → 반복이라는 같은 절차를 따릅니다.

역할 하는 일 코드 위치
마스터 라우터 질문을 읽고 단 한 번의 LLM 호출로 분야를 판별. 확신하면 한 전문가에게, 애매하면 사용자에게 되묻고, 어디에도 안 맞으면 범용으로. graph.py:230 route_specialist
전문 에이전트 ×7 chart · eda · dataanalysis · report · dashboard · workflow · query. 각자 자기 도구만 보이므로 남의 분야 일을 아예 못 합니다. specialists.py:66 SPECIALISTS
범용 에이전트 딱 맞는 전문가 키가 하나도 없을 때의 만능 대체자. graph.py:20 SYSTEM_PROMPT
공통 상태 머신 모든 에이전트가 공유하는 도구 실행 루프 (3절에서 자세히). graph.py:68 build_agent

산출물(artifact)과 artifact_id에 대해서는 5절에서 다룹니다.

여기서 설명한 흐름은 채팅 분석 경로입니다. 그 외에도 공통 상태 머신과 별개로 동작하는 세 가지 독립 LangGraph 파이프라인이 있습니다 — 자세한 내용은 6절에서 설명합니다.

2. 마스터 라우터 — 분야 판별

라우터는 무겁지 않습니다. 도구도, 루프도 없이 한 번의 LLM 호출로 질문을 분류하는 route_specialist (graph.py:230) 함수입니다. 시스템 프롬프트 _ROUTER_SYSTEM (graph.py:174)에 few-shot 예시가 들어 있어, 모델은 전문가 키만 (소문자, 설명 없이) 답합니다.

  1. 질문 도착. domain 필드가 비어 있으면(None) "새 분석" 진입으로 보고 라우터를 돌립니다. 화면 전용 어시스턴트가 이미 분야를 지정했으면 라우터는 건너뜁니다 (routes_chat.py:104-105).
  2. 분류. 모델이 키를 한 개 답하면 확신 → 그 전문가로. 두 개 이상을 답하면 진짜로 애매하다는 뜻입니다 (RouteResult.ambiguous, graph.py:210).
  3. 되묻기(HITL). 애매하면 사용자 턴을 저장한 뒤 clarify SSE(Server-Sent Events, 서버→클라이언트 실시간 전송 — 7절) 이벤트 하나만 보내고 종료합니다. 클라이언트는 같은 질문을 clarified=True와 고른 분야로 다시 보냅니다(턴은 중복 저장되지 않음, routes_chat.py:107-128).
  4. 대체. 알려진 키가 하나도 없으면 GENERAL_KEY="general" (specialists.py:154) — 모든 도구를 쥔 범용 에이전트로 떨어집니다 (라우터에서 import: graph.py:243).
왜 되묻나

라우터가 애매할 때 추측하지 않고 사람에게 묻는 것은 의도된 설계입니다. 예: "revenue와 units의 상관관계 분석" 같은 질문은 eda와 dataanalysis 둘 다에 맞아, 라우터가 두 후보를 돌려주고 화면이 사용자에게 고르게 합니다. 헛다리를 짚느니 한 번 묻는 편이 낫다는 원칙입니다.

3. 공통 상태 머신 (계획 → 실행)

핵심 통찰은 하나입니다 — 그래프 모양은 하나뿐이고, "전문가"란 그 그래프에 좁은 시야를 씌운 같은 그래프의 한 단면일 뿐입니다 (graph.py:56 _scope).

전문가든 범용이든 도는 그래프는 똑같습니다. LangGraph로 만든 두 노드짜리 상태 머신 — 공통 상태 머신 — 입니다 (build_agent, graph.py:68): plan(다음에 뭘 할지 결정)과 execute(도구 실행). 진입점은 plan이고 (graph.py:168), plan이 도구 호출을 내면 execute로, 아니면 END로 갑니다. execute 노드가 실행한 결과를 관찰(observation)이라 하며, 이것이 messages에 쌓여 다음 plan에 입력됩니다. 그래프는 AgentDeps(deps.py:25) 의존성 묶음을 주입받아 실행됩니다 — DB·벡터DB·큐 등 외부 시스템 접근 통로이며, 5절의 AgentDeps에서 자세히 설명합니다.

그래프는 각 스텝마다 AgentState(state.py:13–28)를 공유합니다 — principal · thread_id · messages · artifacts · next_tool · answer 필드로 구성됩니다.

plan 노드는 한 번의 스트리밍 호출로 두 역할을 겸합니다 (graph.py:75-126): 최종 답변일 땐 토큰을 실시간으로 흘려보내고, 도구 호출일 땐 그 조각을 모아 조립합니다. 도구 호출도 내용도 없는 빈 응답이 나오면 chat()로 한 번 재시도합니다 (graph.py:121).

3a. force_tool — 도구 강제 실행 방식

force_tool이 True인 전문가(chart·eda·dataanalysis·report·dashboard·workflow)는 plan 노드의 첫 실행 — 아직 도구 관찰(tool 역할 메시지)이 없는 상태 — 에서 tool_choice="required"로 LLM을 호출해 반드시 도구를 부르게 합니다 (graph.py:94-104). 도구 호출이 오면 즉시 next_tool="execute"로 설정하고 스트리밍을 건너뜁니다. 모델이 도구 호출을 무시하면 일반 스트리밍 경로로 폴백됩니다.

이 강제 구동이 루프와 맞물리는 방식: 첫 plan이 execute로 분기되면 observe 결과가 messages에 누적되고, 다음 plan 실행 시에는 이미 도구 관찰이 존재하므로 tool_choice 강제가 풀려 자유 선택 모드로 전환됩니다. 그래서 확인 문장 같은 자연어 응답은 두 번째 이후 plan에서 정상적으로 스트리밍됩니다.

query만 예외로 force_tool=False입니다 (specialists.py:132-138). 스키마 조회 → SQL 실행으로 여러 도구를 정당하게 연쇄하거나, 산출물 없이 설명만 할 수도 있어서 첫 발을 강제하면 오히려 틀리기 때문입니다.

4. 일곱 전문 에이전트와 범용 에이전트

각 전문가는 위의 3절에서 정의한 plan→execute 루프와 3a절의 force_tool 플래그를 그대로 재사용합니다.

전문가 하나는 그저 (시스템 프롬프트 + 도구 부분집합 + 라우터가 읽는 한 줄 설명)입니다 (Specialist dataclass, specialists.py:25). 같은 공통 상태 머신을 그대로 재사용하고, 분야마다 프롬프트와 보이는 도구만 바꿉니다. 그래서 chart 전문가는 데이터분석 도구를 물리적으로 호출할 수 없습니다 — 그 도구가 목록에 없으니까요.

edit_artifact — 모든 전문가에게 자동으로 추가되는 편집 도구 (_spec(), specialists.py:54, :61). artifact — 에이전트가 생성하는 저장 가능한 결과물, 고유 artifact_id를 가짐 — 에 대한 자세한 패턴은 5절 참조.

아래 표는 일곱 전문가와 편의상 범용 에이전트를 함께 나열합니다. 범용은 어느 전문가 키도 매칭되지 않을 때 사용되는 별도 분류입니다.

전문가 도구
chart schema_retriever, run_sql, build_chart
eda run_eda, profile_dataset
dataanalysis run_data_analysis

비유

전문가에게 도구 부분집합만 쥐여 주는 건, 요리사에게 자기 코스 재료만 담긴 도마를 주는 것과 같습니다. 파티시에 도마에는 생선이 없으니 실수로 생선 요리를 시작할 수가 없습니다 — 실수의 여지 자체를 없앤 설계입니다.

5. 도구와 산출물(artifact)

도구 개요

5절에서는 3절의 execute 노드가 호출하는 도구들과, 에이전트가 산출물(artifact)을 생성·편집하는 구체적인 패턴을 살펴봅니다. 도구는 총 13개(TOOL_SCHEMAS, tools.py:25; 디스패치 표 TOOL_FNS, tools.py:767-781)이며 크게 세 부류로 나뉩니다. artifact_id와 edit_artifact의 동작은 아래 5a에서 설명합니다.

부류도구하는 일
탐색·조회 schema_retriever, run_sql, open_connection, doc_search, profile_dataset 스키마 하이브리드 검색, 읽기 전용 SQL 실행(ODBC 또는 기본 웨어하우스), 문서 KB 검색, 업로드 파일 EDA.
빌더 build_chart, run_eda, run_data_analysis, build_report, build_dashboard_artifact, build_workflow, build_dashboard 스펙을 생성·검증하고 저장 (아래 5a 참조). (build_dashboard만 1급 대시보드 엔티티)
편집 edit_artifact 이미 있는 라이브러리 아티팩트를 자연어로 제자리 수정 (LibraryBuilderService.nl_edit).
비동기 도구

run_eda·run_data_analysis는 작업을 Celery 큐에 넣고 _await_job() (tools.py:719)가 0.5초마다, 최대 60초까지 폴링합니다. 마감까지 안 끝나면 status="running"과 job_id를 담은 running 참조를 돌려줘 클라이언트가 이어서 폴링합니다. profile_dataset은 ydata 기반 전체 HTML EDA 작업을 Celery 큐에 넣고, 폴링은 별도 메커니즘에 맡깁니다.

library_ref — "다시 고칠 수 있는 손잡이"

빌더 도구가 반환하는 Artifact 객체는 kind(결과물 유형)와 payload(세부 데이터)의 두 필드로 구성됩니다. 빌더가 결과를 저장한 뒤에는 _library_ref() (tools.py:444-459)로 이미 저장된 아티팩트를 Artifact(kind=ArtifactKind.LIBRARY_REF, payload={artifact_type, artifact_id, name, status, spec})로 감싸 돌려줍니다. 여기 담긴 artifact_id가 이후 edit_artifact로 같은 결과를 제자리에서 고치게 하는 손잡이입니다. 그래서 "그 차트 선 그래프로 바꿔줘"는 새로 만들지 않고 그 id를 집어 편집합니다.

5b. 산출물 검수 — 저장 전 결정적 검사

빌더 도구는 스펙을 저장하기 전에 세 가지 결정적(LLM 없는) 검사를 거칩니다. 형식(Pydantic)만 맞으면 통과시키던 기존 방식은, 구조는 맞지만 의미가 틀린 산출물을 그대로 저장했습니다 — 예를 들어 존재하지 않는 컬럼을 y축에 지정한 차트는 형식 검증을 통과하고 빈 차트로 렌더된 뒤 대시보드·리포트로 연쇄 전파됩니다. build_chart (tools.py)는 이제 소스의 실제 컬럼을 먼저 해석한 뒤 검사합니다.

비유

출고 검수와 같습니다. 상자 모양이 규격에 맞는지(형식 검증)만 보던 것을, 이제 안에 든 물건이 주문서와 일치하는지까지 봅니다. ① 주문한 부품 번호가 실제 카탈로그에 있는지(없으면 반송), ② 조립 방식이 위험하지 않은지, ③ 재료에 하자는 없는지 — ②·③은 반송이 아니라 "주의" 스티커를 붙여 내보냅니다.

LLM으로 답변을 한 번 더 심사하는 방법(critic)도 있지만, 호출당 수 초의 지연과 비결정성(같은 입력에 다른 판정)이 따릅니다. 그래서 이미 갖춘 sqlglot 분석기·프로파일러로 잡을 수 있는 것은 결정적으로 먼저 잡고 — 0ms, 무료, 재현 가능 — LLM은 남겨 둡니다.

검사성격무엇을 잡나코드
① 컬럼 존재 차단 x·y·color·z·columns·value가 소스 스키마에 실재하는지. 없으면 ChartSpecError로 저장 거부. validate_column_bindings (charts/spec.py)
①.5 SQL 의미 경고 조인 팬아웃(다중 테이블 + 집계 + GROUP BY 없음 → 조용한 중복 집계) 탐지. sql_semantic_warnings (output_checks.py)
③ 데이터 품질 경고 결측률 ≥5% 컬럼 탐지 (예: "discount 8.4% 결측"). missing_value_warnings (output_checks.py)

비차단 경고는 _library_ref()가 payload.quality_warnings에 실어 반환하고, ArtifactView.tsx의 LibraryRefView가 결과 위에 amber 배너로 렌더합니다 — 산출물은 정상 저장·표시되되 사용자가 신뢰 전에 참고할 신호를 남깁니다. LLM critic(②)은 지연·비결정성 때문에 이 범위에서 제외했습니다.

검증은 왜 결정적 우선인가

답변을 LLM으로 한 번 더 심사(critic)하는 방식도 있지만, 검사자가 생성자와 같은 착각을 공유할 수 있고(같은 모델·같은 오해), 호출당 수 초의 지연과 비결정성(같은 입력에 다른 판정)이 따릅니다. 그래서 이 시스템은 결정적으로 잡을 수 있는 건 결정적으로 먼저 잡습니다 — sqlglot AST 분석·스키마 대조·EXPLAIN은 0ms, 무료, 재현 가능하며 회귀 테스트로 고정됩니다. LLM 판정은 결정적 검사가 다 통과했는데도 남는 순수 의미 의심에만 남겨 둡니다.

이렇게 보면 검증은 한 곳이 아니라 여러 층에 걸쳐 있습니다. 아래는 시스템 전체의 검증 지점을 신뢰도·비용 순으로 정리한 것입니다 — 이 문서 곳곳에서 설명한 장치들이 하나의 계층 방어(defense in depth)로 맞물립니다.

층무엇을 검증성격위치
구조 검증 스펙이 형식(필드·타입·그리드·중복 id)에 맞는지 — Pydantic extra="forbid". 결정적 · 차단 validate_chart_spec / validate_report_spec / validate_builder_spec
의미 검증 ① 차트가 바인딩한 컬럼이 소스 스키마에 실재하는지. 결정적 · 차단 validate_column_bindings (5b)
SQL 의미 ①.5 · 데이터 품질 ③ 조인 팬아웃(중복 집계)·결측률 등 위험 신호. 결정적 · 경고 output_checks.py (5b)
실행 검증 생성한 SQL이 실제로 파싱·실행 가능한지 — 실패 시 되먹여 재시도. 결정적 · 자기수정 text2sql EXPLAIN 루프 · nl_graph repair ≤3 (6절)
도구 오류 되먹임 도구가 던진 예외를 관찰로 되먹여 모델이 스스로 교정. 런타임 · 자기수정 execute 노드 (graph.py:155, 3절)
사람 게이트(HITL) 애매한 라우팅·다단계 계획·데이터 품질 이슈를 사람이 승인/선택. 사람 개입 clarify (2절) · planner interrupt · reflect (6절)

아래로 갈수록 비용·불확실성이 커집니다. 그래서 위쪽(구조·의미·실행)에서 결정적으로 최대한 걸러내고, 사람 개입은 진짜 판단이 필요한 지점에만 둡니다. LLM으로 답변 전체를 재심사하는 critic은 가장 마지막 수단이며 현재는 도입하지 않았습니다.

AgentDeps — 외부 시스템 접근 통로

도구는 AgentDeps(deps.py:25)를 통해서만 외부 시스템에 접근합니다. AgentDeps는 도구가 DB·벡터DB·큐 등 외부 시스템에 접근하는 유일한 통로로, 테스트 시 가짜 구현으로 교체할 수 있습니다: llm · query · vector · blob · queue · dashboards · sandbox · artifacts (필수) · odbc · profiler · text2sql · analyzer · run_store (Optional). 예컨대 run_sql은 odbc 핸들이 제공되면 ODBC 커넥터로, 없으면 기본 웨어하우스 엔진으로 읽기 전용 SQL을 실행합니다.

6. 독립 하위 파이프라인 (서비스·라우트 직접 호출)

공통 상태 머신(3절) 안의 도구와 별개로, 그 자체가 독립된 LangGraph인 세 하위 파이프라인이 있습니다. 각자 자기 진입 함수로 컴파일되며, 감독 에이전트가 아니라 해당 서비스/라우트에서 직접 호출됩니다.

이 파이프라인들이 감독 에이전트 도구로 흡수되지 않고 독립된 이유는 세 가지입니다. 첫째, 채팅이 아닌 별도 진입점(데이터프렙 화면, 소스 등록 API 등)에서도 직접 호출됩니다. 둘째, 각자 고유한 검증·자기수정 루프(EXPLAIN 재시도 / repair ≤3 / reflect)를 가지며, 이 다단계 반복을 단일 도구 호출 안에 넣기엔 구조적으로 맞지 않습니다. 셋째, 에이전트 레이어보다 먼저 만들어진 독립 서비스로서 자체 그래프 컴파일·체크포인트를 유지합니다.

아래 그림은 세 파이프라인의 노드 구조를 나란히 보여줍니다.

text2sql

자연어 질문을 검증된 Trino SQL로 변환합니다. 5절의 도구 목록에 없는 이유는 routes_text2sql.py / Text2SQL 서비스가 독립 파이프라인으로 직접 호출하기 때문입니다 (text2sql/graph.py:130 build_text2sql_graph). 라우터는 스키마 매칭 신뢰도 τ(0–1 스코어)를 기준으로 높음(기존 SQL 재사용) · 중간(하이브리드 생성) · 낮음(완전 생성) 세 경로로 분기합니다.

nl_graph

데이터프렙 화면은 NL 입력을 받아 DataPrep GraphSpec DAG — 데이터 변환 노드와 엣지를 JSON으로 기술한 명세 — 를 생성합니다. pick_tables → generate → compile → route → repair(≤3회) 순서로 진행하며, repair 루프는 compile 단계의 검증 오류를 LLM이 수정하는 자기수정 구조입니다 (nl_graph/graph.py:164 build_nl_graph_agent).

column_select

데이터 소스 등록 시 자동 호출됩니다. 판정 결과는 검색 필터 UI의 드롭다운 항목으로 쓰입니다. 내부적으로 reflect 루프(최대 1회)를 돌며 초기 분류 결과를 재검토하고, 카디널리티가 500을 초과하는 컬럼은 하드캡으로 제외합니다. 최종 decide 노드가 색인 대상 컬럼 목록을 확정합니다 (column_select/graph.py:90).

7. 스트리밍과 지속성

답변은 버퍼링 없이 SSE(Server-Sent Events)로 흐릅니다 (EventSourceResponse, routes_chat.py:179-244). 공통 상태 머신을 stream_mode=["updates","custom"] 두 모드로 동시에 구독합니다 (routes_chat.py:198):

  • custom — plan 노드가 밀어내는 answer_token 조각 → answer(token) 이벤트(토큰 단위 실시간 답변).
  • updates — 노드 상태 diff → 산출물마다 artifact, 노드 전환마다 status 이벤트.

전체 이벤트: routed · clarify · answer(token) · artifact · status · done · error. routed는 화면 상단에 "→ 차트 전문가" 같은 표시를 그릴 때 쓰입니다. done은 그래프 정상 종료 시 마지막으로 전송되고, error는 복구 불가 예외 발생 시 전송됩니다.

대화 상태는 Postgres에 지속

LangGraph 대화 상태는 PostgresCheckpointer (adapters/checkpoint/postgres_checkpointer.py:19)가 AsyncPostgresSaver(psycopg_pool.AsyncConnectionPool, autocommit=True, row_factory=dict_row, prepare_threshold=0)로 저장합니다. 팩토리가 앱 시작 시 Container.open_checkpointer() (factory.py:444)로 열고, checkpointer_saver 프로퍼티 (factory.py:469)를 build_agent에 넘깁니다 (routes_chat.py:131). 체크포인트 키는 부서·유저·스레드로 스코프됩니다: {department}:{user_id}:{thread_id} (routes_chat.py:135). 서버가 재시작돼도 대화가 이어지고, 체크포인트가 없으면 채팅 이력을 미리 불러와 메웁니다.

비유

체크포인터는 게임의 자동 저장과 같습니다. 매 턴 상태를 디스크에 적어 두니, 서버가 껐다 켜져도 방금 하던 대화의 그 자리에서 정확히 이어집니다.

8. Context 관리 (컨텍스트 창 방어)

대화가 길어질수록 plan 노드가 LLM에 보내는 메시지가 계속 불어납니다. 이걸 방치하면 모델의 컨텍스트 창(gpt-4o 기준 약 128K 토큰)을 넘어 요청이 실패하고, 같은 이력을 계속 다시 보내므로 그 스레드는 영구히 먹통이 됩니다. 그래서 세 겹의 방어로 컨텍스트를 항상 예산 안에 가둡니다 (core/agent/context.py).

비유

여행 가방을 떠올리세요. 짐이 계속 늘면 결국 안 닫힙니다. 그래서 ① 물건 하나하나를 압축팩에 넣고(큰 도구 결과 잘라내기), ② 최근에 쓴 것만 위 칸에 꺼내 두고 오래된 건 아래로 내리고(최근 우선 트림), ③ 아주 오래된 짐은 목록표 한 장으로 요약해 대체합니다(롤링 요약). 원본은 창고(아티팩트 저장소)에 그대로 있으니 잃어버리는 게 아니라 가방에 넣는 양만 줄이는 것입니다.

왜 필요한가 — 문제의 근원

공통 상태 머신의 messages는 누적(append) 리듀서라 턴마다 늘기만 합니다(3절, state.py:22). plan 노드는 매 호출에서 [시스템 프롬프트, *state["messages"]] 전부를 LLM에 넘겼고, execute 노드는 도구 결과를 str(artifact.payload)로 통째 저장했습니다 — SQL 전체 행이나 EDA 결과 하나가 수천~수만 토큰을 차지합니다. 토큰 세기·자르기·요약이 전혀 없어, 도구를 많이 쓰는 긴 대화는 창을 넘겨 하드 에러로 끝났습니다. 이 절의 방어가 그 실패 경로를 없앱니다.

① 도구 결과 절단 — 가장 큰 누수부터 (P0)

execute 노드는 이제 도구 결과를 summarize_observation() (context.py)로 감쌉니다: 컴팩트 JSON으로 직렬화하고(파이썬 str() 대신 — 무손실 10~30% 절감), 표 모양 결과는 상위 agent_tool_result_max_rows(기본 50)행만 남기고 나머지는 "N행 중 50행" 표기로 대체하며, 전체 길이가 agent_tool_result_max_chars(기본 6000)를 넘으면 잘라냅니다. 원본 데이터는 아티팩트 저장소에 그대로 있으니, 여기서 줄이는 건 모델이 매 턴 다시 읽는 양뿐입니다.

② 롤링 요약 — 오래된 턴 보존 (P2, 선택)

agent_context_summarize가 켜져 있고 이력이 agent_context_summarize_threshold를 넘으면, plan은 최근 agent_context_keep_recent개를 뺀 오래된 턴을 LLM 한 번으로 누적 요약(progressive)해 하나의 system 메시지로 접습니다 (_summarize_turns (graph.py)). 이전 요약 위에 새 요약을 쌓아 이미 압축한 이력을 다시 인코딩하지 않고, 요약 결과는 state.summary에 저장돼 다음 턴에 이어집니다. 요약 호출이 실패해도 이전 요약을 유지해 턴 자체는 절대 깨지지 않습니다(트림이 여전히 창을 지킴). 트림보다 정보 보존이 낫지만 LLM 호출 비용이 들어 기본은 꺼져 있습니다.

③ 토큰 예산 트림 — 하드 실패 차단 (P1)

plan은 LLM 호출 직전 이력을 trim_history() (context.py)로 agent_context_max_tokens(기본 96K) 예산 안에 맞춥니다: 최근 우선으로 남기고, 앞의 시스템 프롬프트는 항상 지키며, 윈도가 도구 결과만 홀로 시작하거나 결과 없는 tool_call로 끝나지 않도록 턴 경계에서 자릅니다. 짝이 깨진 도구 메시지는 OpenAI가 400으로 거절하기 때문에, 이 짝 보존이 트림의 핵심 불변식입니다. 토큰은 문자 기반 근사(≈4자/토큰)로 세어 core 순수성을 지킵니다.

설정 (settings.py)기본값역할
agent_context_max_tokens96,000LLM 전송 이력의 토큰 예산(트림 기준)
agent_context_summarizefalse오래된 턴 롤링 요약 on/off
agent_context_summarize_threshold80,000이 토큰 초과 시 요약 트리거

이 값들은 factory.py agent_deps()에서 ContextConfig로 묶여 AgentDeps.context로 주입됩니다 — core는 settings를 직접 읽지 않습니다(2절 헥사고날 원칙). 트림이 실제로 발생하면 agent.context 로거가 경고를 남겨 관측 가능합니다(P3).

9. 정리

질문이 POST /chat/ask로 들어오면 마스터 라우터가 분야를 판별해 전문가(또는 범용) 에이전트를 선택합니다. 선택된 에이전트는 공통 상태 머신의 plan → execute 루프를 돌며 도구를 호출하고 산출물을 생성하며, 빌더 산출물은 저장 전 결정적 검수(컬럼 존재 차단 · SQL 의미/품질 경고)를 거칩니다(5b절). 그 결과는 SSE 스트림으로 토큰 단위로 클라이언트에 전달되고 Postgres 체크포인터가 대화 상태를 지속합니다. 대화가 길어져도 세 겹의 컨텍스트 방어(도구 결과 절단 · 토큰 예산 트림 · 롤링 요약)가 창 초과를 막습니다(8절). 채팅 흐름과 별개로, text2sql · nl_graph · column_select 세 하위 파이프라인은 해당 서비스에서 직접 호출됩니다(6절).

이 문서는 에이전트 오케스트레이션에 집중합니다. 포트/어댑터, 동기·비동기, 인프라(Postgres·Redis·Weaviate·DuckDB·Trino) 구조는 시스템 아키텍처 문서를 참고하세요.
