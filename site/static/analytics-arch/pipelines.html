분석 Agent가 SQL을 찾고 만드는 방법

평범한 한국어로 질문하면, 신뢰할 수 있는 SQL 코드를 검색 및 작성 해 줍니다. 뒤에서는 세 가지 일이 맞물려 돌아갑니다.
인덱싱은 팀이 이미 써 온 검증된 쿼리·정의·스키마·값을 미리 정리해 네 개의 검색 인덱스로 만들어 둡니다.
검색은 질문이 들어오면 그 인덱스들을 검색하여 관련 쿼리·정의·스키마·값을 찾아 냅니다.
자가 개선은 사용이 축적 되며 다음 검색을 더 낫게 만듭니다.

세 가지 약속.
① 팀의 실제 쿼리·정의에 근거합니다.
② 정의가 확정된 지표(metric)는 AI가 아니라 규칙으로 계산해 매번 같은 숫자를 보장합니다.
③ 기본 설정에서는 어떤 쿼리도 당신이 승인하기 전엔 실데이터에 실행되지 않습니다. (선택 기능인 실행 기반 후보 선택을 켜면, 소량의 샘플 데이터만 읽어서 조회가 일어납니다 — §3 검색 단계 참조.)

💡 점선 밑줄 용어에는 설명이 딸려 있습니다.  ·  각 절 아래엔 딥다이브 박스가 접혀 있습니다. 개념만 읽어도 완결되고, 코드가 궁금하면 그 자리에서 펼치면 됩니다.

1. 큰 그림

데이터 팀의 가치 대부분은 이미 작성된 SQL 안에 잠겨 있습니다. "지역별 분기 매출", "활성 엔터프라이즈 계정" 같은 검증된 쿼리 수백 개가 노트북과 대시보드, 사람들의 머릿속에 흩어져 있죠. 이 플랫폼은 그 흩어진 지식을 누구나 질문만으로 꺼내 쓸 수 있는 엔진으로 바꿉니다. 이를 위해 세 파이프라인이 서로 다른 시점에 돌아갑니다.

파이프라인 시점 하는 일
인덱싱 사전 (offline) 기존 쿼리·정의·스키마·값을 읽어 네 개의 검색 인덱스로 정리·색인. (§2)
검색 실시간 (질문할 때) 네 인덱스를 정해진 순서로 꺼내 질문을 실행 가능한 SQL로 변환. (§3)
자가 개선 사후 (사용 후) 재사용·실행 이력을 되먹여 순위를 조정하고, 사람 승인으로 인덱스를 키움. (§4)

두 저장소, 두 역할

저장소 역할 비유
PostgreSQL 원본 기록(system of record) — 모든 쿼리·정의·승인 상태·사용 통계의 영구 원본. 도서관 마스터 아카이브. 절대 사라지지 않는 원본.
Weaviate 벡터 검색 인덱스 — "비슷한 것 찾아줘"에 최적화된 빠른 사본. 용도별 4개 컬렉션(§2). 유실돼도 원본에서 다시 만듭니다. 카드 목록함. 아카이브를 가리키는 포인터.

진짜 쿼리는 PostgreSQL에 보관 되고, 검색용 색인 정보만 Weaviate에 삽니다. 그래서 색인 정보가 통째로 사라져도 원본에서 다시 만들 수 있습니다.

2. 인덱싱 — 네 개의 인덱스

목표는 하나입니다. 검색이 나중에 꺼내 쓸 지식을 미리 "찾을 수 있게" 만드는 것. 이를 위해 네 종류의 지식을 각기 다른 소스에서 가져와 Weaviate의 네 컬렉션으로 색인합니다.

네 인덱스의 공통점은 이렇습니다:
1. 모두 Weaviate 컬렉션이고,
2. text2vec-transformers가 서버에서 자동으로 벡터화하며,
3. department 필드로 부서끼리 서로 안 보이게 격리되고,
4. 원본이 사라져도 다시 만들 수 있습니다.

차이는 무엇을 벡터로 바꾸느냐(임베딩하는 글의 구성) 입니다. 바로 이 차이가 그 인덱스의 검색 성격을 결정합니다.

인덱스별 정의

인덱스 소스 (어디서) 담는 것 (색인 문서) 검색에서의 역할
query_corpus 팀이 제출한 검증된 SQL. 사람이 직접 채우는 유일한 인덱스. 설명 + 코멘트 + 테이블 + measure 별칭 + 정규형 SQL. 질문과 비슷한 기존 쿼리를 찾아 재사용하거나 few-shot 예시로 씀. (§3 retrieve)
semantic_catalog 승인된 코퍼스에서 구조적으로 캐낸 온톨로지. measure · dimension · window metric. 하나당 한 문서. 정의된 지표를 규칙으로 결정론적 계산할 후보를 제안. (§3 generate)
schema_catalog 웨어하우스 스키마를 Trino로 훑어 색인. 테이블당 한 문서 — FQN · 설명 · 컬럼. 질문에 맞는 테이블·컬럼을 좁혀 줌(schema linking). (§3 generate)
entity_values 범주형 컬럼의 실제 셀값. PII 게이트를 통과한 안전한(safe) 값만. 값 하나가 문서 하나. 질문 속 표현("samsng")을 실제 WHERE 값('Samsung…')에 이어 줌(value linking). (§3 두 경로 공통)

온톨로지·값 연결·쿼리 코퍼스·스키마 카탈로그는 따로 있는 개념이 아니라 바로 이 네 인덱스입니다. 각각을 어떻게 채우는지는 아래 단계 설명에서, 검색에서 언제 쓰는지는 §3에서 다룹니다.

색인 순서도

네 인덱스는 채워지는 계기가 서로 다릅니다. query_corpus는 사람이 직접 제출하고, semantic_catalog는 승인된 코퍼스에서 파생되며, schema_catalog는 웨어하우스를 훑어 만들고, entity_values는 컬럼 판정과 PII 게이트를 거칩니다. 아래는 각 인덱스가 채워지는 경로입니다. 공통점은 하나 — 모두 사람 승인이나 자동 게이트를 지나야 검색에 노출된다는 점입니다.

▶ query_corpus 사용자 제출 쿼리 → 코퍼스 색인 사람이 직접 채우는 유일한 인덱스

흐름은 세 단계입니다. 제출된 쿼리를 한 번 분석해 여러 산출물로 쪼개고, 그 전부를 PostgreSQL에 원본으로 저장한 뒤, 검색에 필요한 것만 Weaviate에 색인합니다. 단, 승인되고 파싱에 성공한 쿼리만 색인됩니다.

승인은 기본적으로 자동이 아닙니다(corpus_auto_approve=False). 제출된 쿼리는 pending 상태로 리뷰 큐에 대기하다가, 관리자 승인 후에야 색인·검색에 노출됩니다. 재사용 경로가 "코퍼스에 있음 = 검증됨"으로 믿고 쓰기 때문입니다(§3 재사용). 이미 신뢰하는 쿼리를 대량으로 넣을 때만 auto_approve로 이 게이트를 건너뛸 수 있습니다.

  1. 제출. 자주 쓰는 검증된 SQL을 제목·코멘트(선택)와 함께 올립니다.
  2. 분석. SQL 파서가 한 번에 여러 산출물을 뽑습니다: 정규형(중복·유사도 판정용), 값 자리를 :param_n 빈칸으로 비운 템플릿형, 그리고 테이블·컬럼·역할 정보.
  3. AI 설명. LLM이 SQL과 코멘트를 한 문장으로 요약합니다 — "주어진 연도의 고객 지역별 총 결제액(GMV)." 이 문장이 의미 검색의 다리가 됩니다. (실패해도 그냥 진행합니다.)
  4. PostgreSQL 저장. 원본 SQL, 정규형, 템플릿, 파라미터 정보, 테이블·컬럼, 설명, 승인 상태, 사용 통계를 각 컬럼에 나눠 저장합니다. 이 원본이 있어 검색용 사본이 사라져도 복구할 수 있습니다.
  5. Weaviate 색인. 이제 검색 가능하게 만들 차례입니다. 색인하는 글은 설명 + 코멘트 + 테이블 + measure 별칭 + 정규형 SQL을 줄바꿈으로 이어 붙인 것이고, 이를 text2vec가 벡터로 바꿉니다. 여기서 gmv·net_margin 같은 지표 별칭을 일부러 별도 줄로 넣습니다. 그래야 "margin by region" 같은 질문이 팀의 지표 이름과 바로 맞물립니다. 결과적으로 hybrid search가 가능해집니다. (template_sql·query_id·tables 등은 검색어가 아니라 부가정보로만 함께 저장됩니다.)
▶ semantic_catalog 승인 코퍼스 → 온톨로지 채굴 → 색인 지표 정의가 어떻게 만들어지나 (온톨로지)

온톨로지 정의를 사람이 손으로 쓰지 않습니다. 승인된 SQL 코퍼스에서 구조만 보고(LLM 없이) measure·dimension을 캐낸 뒤, 사람이 승인한 것만 색인합니다. 이 인덱스는 검색 때 정의된 지표를 규칙으로 계산하는 근거가 됩니다(§3 생성).

  • 채굴. SQL 구조를 그대로 읽습니다 — 집계함수는 measure로, GROUP BY는 dimension으로, JOIN은 관계로. 이를 base 테이블별로 묶어 후보(candidate)를 만듭니다. 이때 JOIN이 값을 부풀리는지를 HLL과 정확 집계로 확인해 fan-out을 막습니다.
  • 승인 게이트. candidate는 리뷰 큐로 갔다가 관리자 승인 시 approved가 됩니다. 승인되는 즉시 색인까지 이어집니다 — 이것이 코드 전체에서 유일하게 자동으로 도는 재색인입니다.
  • 색인. measure·dimension을 하나당 한 문서로 만듭니다. 함께 쓰이던 조합(query shape)과 window metric도 같이 색인해, 조합과 순위/이동평균 지표까지 검색 단위로 제공합니다.
왜 규칙 계산인가 — 같은 지표는 매번 같은 숫자

"GMV"·"활성 사용자"처럼 정의가 확정된 지표는 검색 때 AI가 SQL을 새로 쓰지 않고 규칙으로 계산합니다(§3). 그래야 누가 언제 물어도 결정론적으로 같은 값이 나옵니다.

▶ schema_catalog 웨어하우스 스키마 → 크롤 → 색인 테이블·컬럼 메타를 최신으로 (자동)

  • 초기 크롤. Trino에서 스키마를 가져와 테이블당 한 문서로 색인합니다(글 = FQN · 설명 · 컬럼, 초기 설명은 비어 있음). 여기엔 사람 승인이 없습니다. 스키마는 사실이지 의견이 아니기 때문입니다.
  • 주기적 동기화. 백그라운드 sync_schema 잡이 실제 스키마와 대조합니다. 새 테이블은 색인하고, 컬럼이 바뀐 테이블은 LLM이 기존 설명을 지우지 않고 병합하며(뜻을 새로 지어내지 않음), 변화가 없으면 건너뜁니다.

사용자가 쓴 표현("samsng" 등)을 실제 셀값 'Samsung Electronics Co., Ltd.'로 이으려면, 범주형 컬럼의 실제 값을 색인해 둬야 합니다. 단, 아무 값이나 넣지 않고 두 안전장치를 거칩니다. 이 인덱스는 검색에서 값 연결(value linking)의 근거가 됩니다(§3 — 두 경로 공통).

  1. 안전장치 A — 컬럼 선별 agent. 단계형 agent가 "이 컬럼은 사람이 이름으로 부르는 값인가?"를 판정합니다(region=예, user_id=아니오). 먼저 profile이 고유값 개수·샘플을 규칙으로(AI 없이) 모으고, classify가 그 근거 위에서만 LLM으로 판단하며, reflect가 스스로 되짚어 오판을 고칩니다. 고유값이 500을 넘으면 LLM을 부르기도 전에 규칙으로 잘라냅니다 — 값이 너무 많거나 자유롭게 적는 컬럼은 개인정보가 숨기 쉽기 때문입니다.
  2. 대표값 뽑기. 자주 나오는 순으로 최대 200개 값만 담습니다. 흔한 값일수록 질문에 등장할 확률이 높으니까요.
  3. 안전장치 B — PII 게이트. PII 검출기(Presidio)가 safe면 색인, pii면 제외, uncertain이면 색인하지 않고 리뷰 큐로 보냅니다. 즉 애매하면 막는(fail-closed) 방식입니다. A에서 agent가 "값 컬럼"이라 판단했어도 여기서 다시 검사합니다 — A는 필요조건일 뿐 충분조건이 아닙니다.
  4. 값 색인. safe 값은 가공 없이 그대로 넣습니다(값 하나가 문서 하나). 문자열이 정확히·비슷하게 걸릴 확률을 최대로 높이려는 것입니다. 맥락 해석은 다른 인덱스가 맡습니다.

3. 검색 — 질문에서 SQL로

목표는 평범한 한국어 질문을 정확하고 실행 가능한 SQL로 바꾸는 것입니다. 내부는 LangGraph로 짠 상태 머신(6개 노드)이고, 네 인덱스를 정해진 순서로 꺼내 씁니다. 한 가지 짚어 둘 점 — 네 인덱스는 동시에 조회되지 않습니다. 코드상 병렬 실행(asyncio.gather) 없이 하나씩 순서대로, 그리고 경로에 따라 일부만 씁니다.

검색 순서도 — 네 인덱스가 언제 쓰이나

아래 순서도는 어느 노드에서 어느 인덱스가 조회되는지를 색으로 표시합니다. query_corpus는 항상 첫 노드(retrieve)에서, 나머지 셋은 생성 경로에서만 순차로 쓰입니다. 재사용 경로는 corpus + entity_values만 씁니다.

검색 단계 설명

  1. retrieve — 도서관 검색 (query_corpus). 색인된 쿼리에 hybrid search(의미+키워드)를 걸어 관련 있는 상위 쿼리를 가져옵니다. 검색하기 전에 질문 속 전문용어를 glossary에서 부분 문자열 매칭으로 찾습니다. 매칭되면 그 정의를 덧붙여, 원본 질문과 정의를 붙인 질문 두 가지로 각각 검색한 뒤 결과를 합칩니다. 이렇게 하면 벡터 모델이 "GMV"라는 말을 몰라도, 정의가 관련 쿼리를 끌어올립니다. 이 단계가 조회하는 인덱스는 corpus 하나뿐입니다. 나머지 셋은 뒤의 생성 경로에서 쓰입니다. (glossary는 인덱스가 아니라 별도 용어 사전으로, retrieve와 생성 프롬프트 양쪽에서 쓰입니다.)
  2. 인기도는 정렬에만, 경로 판정에는 안 씁니다. 자주 재사용·실행된 쿼리는 비슷한 후보끼리의 순서를 정할 때 작은 가점을 받습니다(§4). 하지만 경로를 가르는 기준값 best_score에는 인기도를 섞지 않습니다. 인기 쿼리가 당신 질문에 더 잘 맞는 쿼리를 밀어내고 재사용으로 새어 나가는 것을 막기 위해서입니다.
  3. route — 갈림길(점수 3구간). 순수 관련도 best_score를 세 구간으로 나눕니다. ≥ τ(0.75)면 재사용만, 회색지대 [0.60, τ)면 hybrid(재사용·생성을 둘 다), < 0.60이면 생성만. 하나의 경계선만 쓰면 0.74와 0.76이 전혀 다른 길로 갈라지는 절벽이 생깁니다. 회색지대는 그 절벽을 없애고, 애매한 점수대에서는 두 전략을 함께 돌려 교차검증합니다. (회색지대는 hybrid_threshold < reuse_threshold로 설정했을 때만 생깁니다.)
  4. 재사용 경로 — 기존 쿼리 재활용. 가장 잘 맞는 쿼리의 템플릿을 불러와 값만 새로 채웁니다. 채우기 전에 그 값을 value linking(entity_values)에 통과시켜 "캘리포니아"→'CA'처럼 실제 셀값에 맞춥니다. 이 과정을 건너뛰면, 문법과 구조가 완벽한데도 결과가 0건인 쿼리가 나올 수 있습니다 — 값이 실제와 안 맞으면 EXPLAIN조차 이를 잡지 못합니다. 값은 자리표시자 바인딩으로 끼워 넣으므로 주입 공격이 불가능하고, 검증된 구조는 그대로 유지됩니다. 
    template:  ... WHERE o.status = :param_1 AND o.year = :param_2 ...
filled:    ... WHERE o.status = 'paid'  AND o.year = 2023 ...
                                          ↑ 값만 바뀜(값 연결로 접지), 구조는 그대로
  5. 생성 경로 — 여러 후보를 만들어 고르기. 맞는 저장 쿼리가 없으면 새로 씁니다. 단 "한 번 써서 끝"이 아니라 근거 준비 → 여러 후보 → 심판의 구조입니다.
    1. ⓐ semantic_catalog (지름길). 질문이 정의된 지표("GMV", "활성 사용자")에 딱 들어맞으면, AI 대신 규칙으로 결정론적 SQL을 만들어 그 자리에서 확정합니다(아래 단계는 건너뜀). 안 맞으면 다음으로 갑니다.
    2. ⓑ schema linking (schema_catalog). 수천 개 컬럼을 전부 AI에 줄 수 없어 강하게 좁힙니다. 세 방향을 씁니다: backward(유사 쿼리가 쓴 테이블), 카탈로그 직접 검색(질문을 schema_catalog에 바로 걸어, 코퍼스와 무관하게 테이블 발견), forward(필요한 최소한만 추림). 이어 후보 테이블의 컬럼 목록도 카탈로그에서 가져옵니다.
    3. ⓒ glossary + value linking (entity_values). 먼저 질문 속 전문용어를 glossary에서 다시 찾아 (retrieve와 별개로) 그 정의를 프롬프트에 넣습니다. 이어 질문과 용어를 entity_values에 걸어, "samsng" 같은 표현을 실제 값 'Samsung Electronics Co., Ltd.'로 이은 값 힌트를 함께 넣습니다.
    4. ⓓ 여러 초안 → 선택. LLM으로 독립 초안을 candidate_n개 만듭니다 (옵션으로, 실행 기반 선택이 켜져 있으면 질문을 잘게 쪼개 푸는 분해(D&C) 초안도 추가). 그중 하나를 이 단계 안에서 고릅니다. 기본은 자기일관성 다수결이고, 옵션을 켜면 실제로 돌려 본 결과로 투표합니다(결과 0건이거나 그럴듯하지만 틀린 SQL을 걸러냄).
    ⓑ·ⓒ는 순서가 아니라 근거를 모으는 준비이고, ⓓ가 실제 후보를 만드는 단계입니다. 코퍼스가 부실해도 카탈로그가 테이블을 직접 대므로, 생성 품질이 코퍼스에만 매이지 않습니다.
  6. hybrid — 회색지대 교차검증. 재사용 후보 하나와 생성 후보 여럿을 한 풀에 모아 같은 선택기로 심판합니다. 템플릿이 없으면 생성만으로, 생성이 비면 재사용만으로 자연스럽게 한쪽으로 물러납니다.
  7. validate — EXPLAIN으로 스스로 교정 (실데이터는 안 건드림). 채택된 SQL을 EXPLAIN으로 검사합니다. 실패하면 LLM이 그 오류를 보고 SQL을 고쳐 최대 N회 다시 검사합니다. 정해진 횟수를 다 쓰면 실패로 마감합니다. (엔진 오류 메시지는 서버에만 기록하고 500자로 잘라 씁니다 — 프롬프트 조작이나 스키마 유출을 막기 위해서입니다.)
  8. explain → 승인 → 실행. LLM이 2~3문장짜리 쉬운 요약을 붙입니다. LangGraph의 역할은 여기까지입니다. 사람의 승인과 실행은 그래프 밖에서 일어나며, 기본 설정에서는 승인 전에 실데이터에 실행되지 않습니다. 실행 이력은 PostgreSQL에 쌓여 다음 검색을 개선합니다(§4).

4. 자가 개선 파이프라인 (학습)

검색은 한 번 만들어 두고 끝나는 조회가 아닙니다. 플랫폼은 쿼리를 보여 주고·재사용하고·실행할 때마다 그 기록을 남기고, 사람이 승인한 새 지식을 인덱스에 편입합니다. 그래서 시간이 지나면 누가 일일이 표시하지 않아도 어떤 쿼리·정의가 팀에 중요한지 저절로 학습됩니다. 되먹임은 두 갈래로 맞물립니다 — ① 사용 신호(자동으로 순위 조정)와 ② 사람 승인(게이트를 통해 인덱스 성장).

① 사용 신호 — 자동 순위 조정

플랫폼은 쿼리를 보여 주고·재사용하고·실행할 때마다 그 횟수를 기록합니다. 외부 데이터가 아니라 우리가 직접 쌓는 신호죠. 자주 쓰인 쿼리는 다음 검색에서 약간의 가점을 받아, 좋은 답이 계속 위로 올라옵니다. 중요한 건 이 가점을 순서를 정할 때만 쓴다는 점입니다. 경로를 가르는 기준값(best_score)에는 섞지 않습니다. 인기만으로 당신 질문에 더 잘 맞는 쿼리를 밀어내는 일을 막기 위해서입니다(§3 route).

② 사람 승인 — 인덱스 성장

새 지식은 사람 승인을 거쳐야 인덱스에 들어옵니다. 승인 경로는 두 갈래이고, 서로 독립적입니다. (a) 코퍼스 제출은 쿼리 행 자체가 pending 상태로 대기하다가 관리자 승인 시 색인됩니다 — enrichment 리뷰 큐를 거치지 않습니다. (b) 채굴 후보(glossary 용어, semantic 모델, PII 불확실 값)는 enrichment_reviews 큐에서 대기하다 승인 후 편입됩니다.

승인 대상 승인 시 동작
corpus 쿼리 query_corpus에 색인 → 재사용·few-shot에 노출.
glossary term 승인 term으로 저장 → 검색 시 find-time 부분 문자열 확장에 사용.
semantic 모델 index_model()이 즉시 semantic_catalog 재색인 — 유일한 자동 재색인 트리거.
PII 불확실 값 사람이 safe/pii 판정 → safe면 이후 entity_values 색인 대상.
주의 — 채굴·재색인은 대부분 수동 실행

코퍼스에 새 쿼리가 승인돼도 semantic_catalog가 저절로 다시 채굴되지는 않습니다. 컬럼 사용·glossary·semantic 채굴은 관리자가 필요할 때 직접 돌리고, 그 결과가 리뷰 큐를 거쳐 승인될 때 색인됩니다. 사람 손 없이 알아서 최신화되는 것은 백그라운드 sync_schema 잡이 도는 schema_catalog뿐입니다(스키마는 사실이라 사람 판단이 필요 없으니까요).

분석 Agent — 내부 문서. 인덱싱은 네 인덱스로 기존 지식을 찾을 수 있게 만들고, 검색은 그 인덱스를 정해진 순서로 꺼내 질문을 믿을 수 있는 SQL로 바꾸며, 자가 개선은 사용 흔적과 사람 승인을 되먹여 다음 검색을 낫게 합니다. 핵심은 셋 다 같습니다 — 잘 되는 것을 재사용하고, 필요할 때만 신중히 새로 만들며, 통제권은 늘 사람에게 둡니다.
