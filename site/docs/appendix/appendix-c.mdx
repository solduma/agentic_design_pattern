# 부록 C: 에이전틱 프레임워크 빠른 개요

## LangChain

LangChain은 LLM 기반 애플리케이션 개발을 위한 프레임워크입니다. 핵심 강점은 LangChain Expression Language(LCEL)로, 컴포넌트를 "파이프"로 연결하여 체인을 구성할 수 있습니다. 이를 통해 한 단계의 출력이 다음 단계의 입력이 되는 명확하고 선형적인 시퀀스가 만들어집니다. Directed Acyclic Graphs(DAG), 즉 루프 없이 한 방향으로만 흐르는 워크플로에 적합하게 설계되었습니다.

**사용 권장 상황:**

- **단순 RAG**: 문서를 검색하고 프롬프트를 생성하여 LLM으로부터 답변을 얻는 경우
- **요약(Summarization)**: 사용자 텍스트를 요약 프롬프트에 전달하고 출력을 반환하는 경우
- **추출(Extraction)**: 텍스트 블록에서 JSON과 같은 구조화된 데이터를 추출하는 경우

```python
# A simple LCEL chain conceptually
# (This is not runnable code, just illustrates the flow)
chain = prompt | model | output_parse
```

## LangGraph

LangGraph는 더 고급 에이전틱 시스템을 처리하기 위해 LangChain 위에 구축된 라이브러리입니다. 노드(함수 또는 LCEL 체인)와 엣지(조건부 로직)로 이루어진 그래프로 워크플로를 정의할 수 있습니다. 주요 장점은 사이클(cycle) 생성 기능으로, 작업이 완료될 때까지 애플리케이션이 루프, 재시도, 또는 유연한 순서로 도구를 호출할 수 있습니다. 노드 간에 전달되고 프로세스 전체에 걸쳐 업데이트되는 애플리케이션 상태를 명시적으로 관리합니다.

**사용 권장 상황:**

- **다중 에이전트 시스템(Multi-agent Systems)**: 슈퍼바이저 에이전트가 전문 워커 에이전트에 작업을 라우팅하며, 목표 달성까지 루프를 반복하는 경우
- **계획-실행 에이전트(Plan-and-Execute Agents)**: 에이전트가 계획을 수립하고 단계를 실행한 뒤 결과를 바탕으로 계획을 업데이트하기 위해 루프로 돌아오는 경우
- **Human-in-the-Loop**: 그래프가 다음 노드를 결정하기 전에 사람의 입력을 기다려야 하는 경우

| 기능 | LangChain | LangGraph |
|---|---|---|
| 핵심 추상화 | 체인 (LCEL 사용) | 노드 그래프 |
| 워크플로 유형 | 선형 (Directed Acyclic Graph) | 순환형 (루프가 있는 그래프) |
| 상태 관리 | 실행당 일반적으로 무상태 | 명시적이고 영속적인 상태 객체 |
| 주요 용도 | 단순하고 예측 가능한 시퀀스 | 복잡하고 동적인 상태 기반 에이전트 |

**어떤 것을 선택해야 할까요?**

- 애플리케이션의 흐름이 명확하고 예측 가능하며 선형적일 때는 **LangChain**을 선택하십시오. A에서 B, B에서 C로 루프 없이 프로세스를 정의할 수 있다면 LCEL을 활용한 LangChain이 적합합니다.
- 애플리케이션이 추론하고 계획하거나 루프 내에서 동작해야 할 때는 **LangGraph**를 선택하십시오. 에이전트가 도구를 사용하고 결과를 반성하며 다른 접근 방식으로 재시도해야 한다면 LangGraph의 순환적이고 상태 기반인 특성이 필요합니다.

```python
# Graph state
class State(TypedDict):
   topic: str
   joke: str
   story: str
   poem: str
   combined_output: str

# Nodes
def call_llm_1(state: State):
   """First LLM call to generate initial joke"""

   msg = llm.invoke(f"Write a joke about {state['topic']}")
   return {"joke": msg.content}
```

```python
def call_llm_2(state: State):
   """Second LLM call to generate story"""

   msg = llm.invoke(f"Write a story about {state['topic']}")
   return {"story": msg.content}

def call_llm_3(state: State):
   """Third LLM call to generate poem"""

   msg = llm.invoke(f"Write a poem about {state['topic']}")
   return {"poem": msg.content}

def aggregator(state: State):
   """Combine the joke and story into a single output"""

   combined = f"Here's a story, joke, and poem about
{state['topic']}!\n\n"
   combined += f"STORY:\n{state['story']}\n\n"
   combined += f"JOKE:\n{state['joke']}\n\n"
   combined += f"POEM:\n{state['poem']}"
   return {"combined_output": combined}

# Build workflow
parallel_builder = StateGraph(State)

# Add nodes
parallel_builder.add_node("call_llm_1", call_llm_1)
parallel_builder.add_node("call_llm_2", call_llm_2)
parallel_builder.add_node("call_llm_3", call_llm_3)
parallel_builder.add_node("aggregator", aggregator)

# Add edges to connect nodes
parallel_builder.add_edge(START, "call_llm_1")
parallel_builder.add_edge(START, "call_llm_2")
parallel_builder.add_edge(START, "call_llm_3")
parallel_builder.add_edge("call_llm_1", "aggregator")
parallel_builder.add_edge("call_llm_2", "aggregator")
parallel_builder.add_edge("call_llm_3", "aggregator")
parallel_builder.add_edge("aggregator", END)
parallel_workflow = parallel_builder.compile()

# Show workflow
display(Image(parallel_workflow.get_graph().draw_mermaid_png()))

# Invoke
state = parallel_workflow.invoke({"topic": "cats"})
print(state["combined_output"])
```

이 코드는 병렬로 동작하는 LangGraph 워크플로를 정의하고 실행합니다. 주요 목적은 주어진 주제에 대한 농담, 이야기, 시를 동시에 생성한 다음 단일 형식 텍스트 출력으로 결합하는 것입니다.

## Google ADK

Google의 Agent Development Kit(ADK)는 여러 상호작용하는 AI 에이전트로 구성된 애플리케이션을 구축하고 배포하기 위한 고수준의 구조화된 프레임워크입니다. 에이전트의 내부 로직을 위한 기본 빌딩 블록을 제공하는 것보다 에이전트 협업 편성을 위한 더 견해적이고 프로덕션 지향적인 시스템을 제공한다는 점에서 LangChain, LangGraph와 차별화됩니다.

LangChain은 가장 기초적인 수준에서 작동하며 모델 호출 및 출력 파싱과 같은 일련의 작업을 생성하기 위한 컴포넌트와 표준화된 인터페이스를 제공합니다. LangGraph는 더 유연하고 강력한 제어 흐름을 도입하여 이를 확장하며, 에이전트의 워크플로를 상태 기반 그래프로 처리합니다. LangGraph를 사용하면 개발자는 노드(함수 또는 도구)와 엣지(실행 경로를 지시)를 명시적으로 정의합니다. 이 그래프 구조는 시스템이 루프를 돌고, 작업을 재시도하며, 노드 간에 전달되는 명시적으로 관리된 상태 객체를 기반으로 결정을 내릴 수 있는 복잡한 순환 추론을 허용합니다. 개발자에게 단일 에이전트의 사고 과정에 대한 세밀한 제어 또는 제1원칙에서 다중 에이전트 시스템을 구성하는 능력을 제공합니다.

Google ADK는 이러한 저수준 그래프 구성의 많은 부분을 추상화합니다. 개발자에게 모든 노드와 엣지를 정의하도록 요구하는 대신, 다중 에이전트 상호작용을 위한 사전 구축된 아키텍처 패턴을 제공합니다. 예를 들어, ADK는 서로 다른 에이전트 간의 제어 흐름을 자동으로 관리하는 SequentialAgent나 ParallelAgent와 같은 내장 에이전트 유형을 보유합니다. 프라이머리 에이전트가 전문화된 서브 에이전트에 작업을 위임하는 에이전트 "팀"의 개념을 중심으로 설계되었습니다. 상태 및 세션 관리는 프레임워크에 의해 더 암묵적으로 처리되어, LangGraph의 명시적 상태 전달보다 더 응집력 있지만 덜 세분화된 접근 방식을 제공합니다. 따라서 LangGraph가 단일 로봇 또는 팀의 복잡한 배선을 설계하기 위한 상세한 도구를 제공하는 반면, Google ADK는 이미 함께 작동하는 방법을 아는 로봇 집단을 구축하고 관리하기 위한 공장 조립 라인을 제공합니다.

## Crew.AI

CrewAI는 협업 역할과 구조화된 프로세스에 집중하여 다중 에이전트 시스템을 구축하기 위한 편성(orchestration) 프레임워크를 제공합니다. 기초 툴킷보다 높은 수준의 추상화에서 작동하며, 인간 팀을 반영하는 개념적 모델을 제공합니다. 로직의 세분화된 흐름을 그래프로 정의하는 대신, 개발자는 행위자와 할당을 정의하고 CrewAI가 그들의 상호작용을 관리합니다.

이 프레임워크의 핵심 컴포넌트는 에이전트(Agents), 태스크(Tasks), 그리고 크루(Crew)입니다. 에이전트(Agent)는 기능뿐만 아니라 페르소나로도 정의되며, 특정 역할, 목표, 배경 스토리를 포함하여 행동과 커뮤니케이션 스타일을 안내합니다. 태스크(Task)는 명확한 설명과 기대 출력을 가진 별개의 작업 단위로, 특정 에이전트에 할당됩니다. 크루(Crew)는 에이전트와 태스크 목록을 포함하는 응집력 있는 단위로, 미리 정의된 프로세스를 실행합니다. 이 프로세스는 워크플로를 지시하며, 일반적으로 순차적(sequential, 한 태스크의 출력이 다음 태스크의 입력이 되는 방식) 또는 계층적(hierarchical, 매니저 역할의 에이전트가 태스크를 위임하고 다른 에이전트들 간의 워크플로를 조율하는 방식) 중 하나입니다.

다른 프레임워크와 비교할 때, CrewAI는 독특한 위치를 차지합니다. 개발자가 모든 노드와 조건부 엣지를 연결하는 LangGraph의 저수준 명시적 상태 관리 및 제어 흐름에서 벗어납니다. 상태 머신을 구축하는 대신, 개발자는 팀 헌장을 설계합니다. Google ADK가 전체 에이전트 수명 주기를 위한 포괄적이고 프로덕션 지향적인 플랫폼을 제공하는 반면, CrewAI는 에이전트 협업 로직과 전문가 팀 시뮬레이션에 특화됩니다.

```python
@crew
def crew(self) -> Crew:
   """Creates the research crew"""
   return Crew(
     agents=self.agents,
     tasks=self.tasks,
     process=Process.sequential,
     verbose=True,
   )
```

이 코드는 AI 에이전트 팀을 위한 순차적 워크플로를 설정하며, 에이전트들이 특정 순서로 태스크 목록을 처리하고 진행 상황 모니터링을 위해 상세 로깅이 활성화됩니다.

## 기타 에이전트 개발 프레임워크

**Microsoft AutoGen**: AutoGen은 대화를 통해 작업을 해결하는 여러 에이전트를 편성하는 것을 중심으로 한 프레임워크입니다. 아키텍처는 고유한 기능을 가진 에이전트들이 상호작용할 수 있게 하여 복잡한 문제 분해와 협업적 해결을 가능하게 합니다. AutoGen의 주요 장점은 동적이고 복잡한 다중 에이전트 상호작용을 지원하는 유연한 대화 중심 접근 방식입니다. 그러나 이 대화적 패러다임은 덜 예측 가능한 실행 경로로 이어질 수 있으며, 작업이 효율적으로 수렴하도록 정교한 프롬프트 엔지니어링이 필요할 수 있습니다.

**LlamaIndex**: LlamaIndex는 근본적으로 대규모 언어 모델을 외부 및 비공개 데이터 소스와 연결하도록 설계된 데이터 프레임워크입니다. RAG(Retrieval-Augmented Generation)를 수행할 수 있는 지식 기반 에이전트를 구축하는 데 필수적인 정교한 데이터 수집 및 검색 파이프라인 생성에 탁월합니다. 컨텍스트 인식 에이전트 생성을 위한 데이터 인덱싱 및 쿼리 기능은 매우 강력하지만, 복잡한 에이전틱 제어 흐름 및 다중 에이전트 편성을 위한 네이티브 도구는 에이전트 우선 프레임워크에 비해 덜 발달되어 있습니다. LlamaIndex는 핵심 기술적 과제가 데이터 검색 및 합성일 때 최적입니다.

**Haystack**: Haystack은 언어 모델 기반의 확장 가능하고 프로덕션 준비된 검색 시스템을 구축하기 위해 설계된 오픈소스 프레임워크입니다. 문서 검색, 질문 응답, 요약을 위한 파이프라인을 형성하는 모듈식 상호운용 가능한 노드로 아키텍처가 구성됩니다. Haystack의 주요 강점은 대규모 정보 검색 작업에서의 성능과 확장성에 대한 집중으로, 기업급 애플리케이션에 적합합니다. 잠재적 트레이드오프는 검색 파이프라인에 최적화된 설계가 매우 동적이고 창의적인 에이전틱 동작을 구현하는 데 더 경직될 수 있다는 점입니다.

**MetaGPT**: MetaGPT는 미리 정의된 표준 운영 절차(Standard Operating Procedures, SOPs) 세트를 기반으로 역할과 작업을 할당하여 다중 에이전트 시스템을 구현합니다. 이 프레임워크는 소프트웨어 개발 회사를 모방하도록 에이전트 협업을 구조화하며, 에이전트가 프로덕트 매니저나 엔지니어 같은 역할을 맡아 복잡한 작업을 완료합니다. SOP 기반 접근 방식은 고도로 구조화되고 일관성 있는 출력을 생성하여, 코드 생성과 같은 특화된 도메인에 상당한 장점을 제공합니다. 프레임워크의 주요 한계는 높은 특화 수준으로 인해 핵심 설계 외의 범용 에이전틱 작업에는 적응성이 낮다는 점입니다.

**SuperAGI**: SuperAGI는 자율 에이전트를 위한 완전한 수명 주기 관리 시스템을 제공하도록 설계된 오픈소스 프레임워크입니다. 에이전트 프로비저닝, 모니터링, 그래픽 인터페이스 등의 기능을 포함하여 에이전트 실행의 신뢰성을 높이는 것을 목표로 합니다. 주요 이점은 루핑과 같은 일반적인 장애 모드를 처리하고 에이전트 성능에 대한 관찰 가능성을 제공하는 내장 메커니즘을 통한 프로덕션 준비성에 있습니다. 잠재적 단점은 포괄적인 플랫폼 접근 방식이 더 경량의 라이브러리 기반 프레임워크보다 더 많은 복잡성과 오버헤드를 초래할 수 있다는 점입니다.

**Semantic Kernel**: Microsoft가 개발한 Semantic Kernel은 "플러그인"과 "플래너" 시스템을 통해 대규모 언어 모델을 기존 프로그래밍 코드와 통합하는 SDK입니다. LLM이 네이티브 함수를 호출하고 워크플로를 편성할 수 있게 하여, 더 광범위한 소프트웨어 애플리케이션 내에서 모델을 추론 엔진으로 효과적으로 처리합니다. 주요 강점은 특히 .NET 및 Python 환경에서 기존 엔터프라이즈 코드베이스와의 원활한 통합입니다. 플러그인 및 플래너 아키텍처의 개념적 오버헤드는 더 간단한 에이전트 프레임워크에 비해 가파른 학습 곡선을 제시할 수 있습니다.

**Strands Agents**: AI 에이전트를 구축하고 실행하기 위한 모델 기반 접근 방식을 사용하는 AWS의 경량 유연한 SDK입니다. 단순하고 확장 가능하도록 설계되어 기본적인 대화형 어시스턴트부터 복잡한 다중 에이전트 자율 시스템까지 모든 것을 지원합니다. 프레임워크는 모델 불가지론적(model-agnostic)이며 다양한 LLM 제공자에 대한 폭넓은 지원을 제공하고, 외부 도구에 쉽게 접근할 수 있도록 MCP(Model Context Protocol)와의 네이티브 통합을 포함합니다. 핵심 장점은 시작하기 쉬운 사용자 정의 가능한 에이전트 루프를 통한 단순성과 유연성입니다. 잠재적 트레이드오프는 경량 설계로 인해 개발자가 더 포괄적인 프레임워크에서 기본 제공할 수 있는 고급 모니터링이나 수명 주기 관리 시스템과 같은 주변 운영 인프라를 더 많이 구축해야 할 수 있다는 점입니다.

## 결론

에이전틱 프레임워크의 환경은 에이전트 로직 정의를 위한 저수준 라이브러리부터 다중 에이전트 협업 편성을 위한 고수준 플랫폼까지 다양한 스펙트럼의 도구를 제공합니다. 기초적인 수준에서 LangChain은 단순하고 선형적인 워크플로를 가능하게 하는 반면, LangGraph는 더 복잡한 추론을 위한 상태 기반 순환 그래프를 도입합니다. CrewAI와 Google ADK와 같은 고수준 프레임워크는 미리 정의된 역할을 가진 에이전트 팀 편성에 초점을 이동시키며, LlamaIndex와 같은 다른 프레임워크는 데이터 집약적 애플리케이션에 특화됩니다. 이러한 다양성은 개발자에게 그래프 기반 시스템의 세분화된 제어와 더 의견이 반영된 플랫폼의 간소화된 개발 사이에서 핵심적인 트레이드오프를 제시합니다. 결과적으로 올바른 프레임워크 선택은 애플리케이션이 단순한 시퀀스, 동적 추론 루프, 또는 관리되는 전문가 팀을 필요로 하는지에 달려 있습니다. 궁극적으로, 이 진화하는 생태계는 프로젝트가 요구하는 정확한 추상화 수준을 선택함으로써 개발자가 점점 더 정교한 AI 시스템을 구축할 수 있도록 지원합니다.
