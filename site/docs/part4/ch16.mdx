# 16장: 리소스 인식 최적화

리소스 인식 최적화(Resource-Aware Optimization)는 지능형 에이전트가 동작 중에 연산·시간·비용 자원을 동적으로 모니터링하고 관리할 수 있도록 합니다. 이는 주로 행동 순서에 집중하는 단순 계획 수립(Planning)과는 다릅니다. 리소스 인식 최적화는 에이전트가 지정된 자원 예산 내에서 목표를 달성하거나 효율을 최적화하기 위해 행동 실행에 관한 결정을 내리도록 요구합니다. 이는 더 정확하지만 비용이 높은 모델과 더 빠르고 저렴한 모델 중 하나를 선택하거나, 더 정교한 응답을 위해 추가 연산을 할당할지 아니면 빠르고 간략한 답변을 반환할지를 결정하는 것을 포함합니다.

예를 들어, 재무 분석가를 위해 대규모 데이터셋을 분석하는 에이전트를 생각해 보십시오. 분석가가 즉시 예비 보고서가 필요하다면, 에이전트는 더 빠르고 저렴한 모델을 사용하여 주요 트렌드를 신속하게 요약할 수 있습니다. 그러나 분석가가 중요한 투자 결정을 위해 높은 정확도의 예측이 필요하고 더 큰 예산과 시간이 있다면, 에이전트는 더 많은 자원을 할당하여 강력하고 느리지만 더 정밀한 예측 모델을 활용합니다. 이 범주에서 핵심 전략 중 하나는 선호하는 모델이 과부하나 속도 제한으로 인해 사용 불가능할 때 안전장치 역할을 하는 폴백(fallback) 메커니즘입니다. 우아한 성능 저하(Graceful Degradation)를 보장하기 위해, 시스템은 완전히 실패하는 대신 서비스 연속성을 유지하며 기본 또는 더 저렴한 모델로 자동 전환합니다.

## 실용적 응용 및 사용 사례

실용적인 사용 사례는 다음과 같습니다.

- **비용 최적화 LLM 사용:** 에이전트가 예산 제약에 따라 복잡한 작업에는 크고 비싼 LLM을, 단순한 쿼리에는 더 작고 저렴한 모델을 사용할지 결정합니다.
- **지연 시간 민감 운영:** 실시간 시스템에서, 에이전트는 시기적절한 응답을 보장하기 위해 더 빠르지만 잠재적으로 덜 포괄적인 추론 경로를 선택합니다.
- **에너지 효율성:** 엣지 디바이스나 제한된 전력이 있는 환경에 배포된 에이전트의 경우, 배터리 수명을 보존하기 위해 처리를 최적화합니다.
- **서비스 신뢰성을 위한 폴백:** 에이전트가 기본 선택이 불가능할 때 자동으로 백업 모델로 전환하여 서비스 연속성과 우아한 성능 저하를 보장합니다.
- **데이터 사용 관리:** 에이전트가 대역폭이나 저장 공간을 절약하기 위해 전체 데이터셋 다운로드 대신 요약된 데이터 검색을 선택합니다.
- **적응형 작업 배분(Adaptive Task Allocation):** 멀티 에이전트 시스템에서, 에이전트가 현재 연산 부하나 가용 시간에 따라 스스로 작업을 할당합니다.

## 실습 코드 예제

사용자 질문에 답하는 지능형 시스템은 각 질문의 난이도를 평가할 수 있습니다. 단순한 쿼리에는 Gemini Flash와 같은 비용 효율적인 언어 모델을 활용합니다. 복잡한 질의에는 더 강력하지만 비용이 높은 언어 모델(Gemini Pro 등)을 고려합니다. 더 강력한 모델 사용 여부는 자원 가용성, 특히 예산 및 시간 제약에도 달려 있습니다. 이 시스템은 적절한 모델을 동적으로 선택합니다.

예를 들어, 계층적 에이전트로 구축된 여행 플래너를 생각해 보십시오. 사용자의 복잡한 요청을 이해하고, 이를 다단계 여정으로 분해하며, 논리적 결정을 내리는 고수준 계획 수립은 Gemini Pro와 같은 정교하고 강력한 LLM이 관리합니다. 이것이 깊은 컨텍스트 이해와 추론 능력이 필요한 "플래너" 에이전트입니다.

그러나 계획이 수립되면, 항공권 가격 조회, 호텔 가용성 확인, 레스토랑 리뷰 찾기와 같은 계획 내 개별 작업들은 본질적으로 단순하고 반복적인 웹 쿼리입니다. 이러한 "도구 함수 호출(tool function calls)"은 Gemini Flash와 같이 더 빠르고 저렴한 모델로 실행할 수 있습니다. 이러한 간단한 웹 검색에는 저렴한 모델을 사용하고, 복잡한 계획 단계에는 일관성 있고 논리적인 여행 계획을 보장하기 위해 더 고급 모델의 뛰어난 지능이 필요한 이유를 직관적으로 이해할 수 있습니다.

Google의 에이전트 개발 키트(ADK, Agent Development Kit)는 모듈형 및 확장 가능한 애플리케이션을 가능하게 하는 멀티 에이전트 아키텍처를 통해 이 접근법을 지원합니다. 서로 다른 에이전트가 전문화된 작업을 처리할 수 있습니다. 모델 유연성은 Gemini Pro와 Gemini Flash를 포함한 다양한 Gemini 모델을 직접 사용하거나 LiteLLM을 통해 다른 모델을 통합할 수 있게 합니다. ADK의 오케스트레이션(Orchestration) 기능은 적응형 동작을 위한 동적 LLM 기반 라우팅을 지원합니다. 내장된 평가 기능은 에이전트 성능의 체계적인 평가를 가능하게 하며, 시스템 개선에 활용할 수 있습니다(평가 및 모니터링 장 참조).

다음으로, 동일한 설정이지만 서로 다른 모델과 비용을 활용하는 두 에이전트를 정의합니다.

라우터 에이전트(Router Agent)는 쿼리 길이와 같은 단순한 지표를 기반으로 쿼리를 안내할 수 있습니다. 짧은 쿼리는 덜 비싼 모델로, 긴 쿼리는 더 유능한 모델로 보냅니다. 그러나 더 정교한 라우터 에이전트는 LLM 또는 ML 모델을 활용하여 쿼리의 뉘앙스와 복잡성을 분석할 수 있습니다. 이 LLM 라우터는 어떤 다운스트림 언어 모델이 가장 적합한지 결정할 수 있습니다. 예를 들어, 사실 검색을 요청하는 쿼리는 플래시 모델로 라우팅되고, 심층 분석이 필요한 복잡한 쿼리는 프로 모델로 라우팅됩니다.

최적화 기법은 LLM 라우터의 효과를 더욱 높일 수 있습니다. 프롬프트 튜닝은 라우터 LLM이 더 나은 라우팅 결정을 내리도록 프롬프트를 제작하는 것입니다. 쿼리와 최적 모델 선택의 데이터셋으로 LLM 라우터를 파인튜닝(fine-tuning)하면 정확도와 효율성이 향상됩니다. 이 동적 라우팅 기능은 응답 품질과 비용 효율성의 균형을 맞춥니다.

```python
# Conceptual Python-like structure, not runnable code

from google.adk.agents import Agent, BaseAgent
from google.adk.events import Event
from google.adk.agents.invocation_context import InvocationContext
import asyncio

class QueryRouterAgent(BaseAgent):
   name: str = "QueryRouter"
   description: str = "Routes user queries to the appropriate LLM
agent based on complexity."

   async def _run_async_impl(self, context: InvocationContext) ->
AsyncGenerator[Event, None]:
       user_query = context.current_message.text # Assuming text
input
       query_length = len(user_query.split()) # Simple metric: number
of words

       if query_length < 20: # Example threshold for simplicity vs.
complexity
           print(f"Routing to Gemini Flash Agent for short query
(length: {query_length})")
           # In a real ADK setup, you would 'transfer_to_agent' or
directly invoke
           # For demonstration, we'll simulate a call and yield its
response
           response = await
gemini_flash_agent.run_async(context.current_message)
           yield Event(author=self.name, content=f"Flash Agent
processed: {response}")
       else:
           print(f"Routing to Gemini Pro Agent for long query
(length: {query_length})")
           response = await
gemini_pro_agent.run_async(context.current_message)
           yield Event(author=self.name, content=f"Pro Agent
processed: {response}")
```

비평 에이전트(Critique Agent)는 언어 모델의 응답을 평가하여 여러 기능을 수행하는 피드백을 제공합니다. 자기 수정(Self-Correction)을 위해 오류나 불일치를 식별하여 응답 에이전트가 출력을 개선하도록 유도함으로써 품질을 향상시킵니다. 또한 정확도와 관련성 같은 지표를 추적하는 성능 모니터링을 위해 응답을 체계적으로 평가하며, 이는 최적화에 활용됩니다. 또한 피드백은 강화학습이나 파인튜닝(fine-tuning)의 신호가 될 수 있습니다. 예를 들어, 플래시 모델 응답이 부적절하다는 일관된 식별은 라우터 에이전트의 로직을 개선할 수 있습니다. 직접적으로 예산을 관리하지는 않지만, 비평 에이전트는 단순한 쿼리를 프로 모델로 보내거나 복잡한 쿼리를 플래시 모델로 보내는 것과 같은 최적화되지 않은 라우팅 선택을 식별함으로써 간접적인 예산 관리에 기여하며, 이는 좋지 않은 결과로 이어집니다. 이는 자원 배분과 비용 절감을 개선하는 조정을 알립니다.

비평 에이전트는 응답 에이전트가 생성한 텍스트만을 검토하거나, 원래 쿼리와 생성된 텍스트 모두를 검토하도록 구성할 수 있어, 최초 질문과의 응답 정렬에 대한 포괄적인 평가를 가능하게 합니다.

```text
CRITIC_SYSTEM_PROMPT = """
You are the **Critic Agent**, serving as the quality assurance arm of
our collaborative research assistant system. Your primary function is
to **meticulously review and challenge** information from the
Researcher Agent, guaranteeing **accuracy, completeness, and unbiased
presentation**.
Your duties encompass:
* **Assessing research findings** for factual correctness,
thoroughness, and potential leanings.
* **Identifying any missing data** or inconsistencies in reasoning.
* **Raising critical questions** that could refine or expand the
current understanding.
* **Offering constructive suggestions** for enhancement or exploring
different angles.
* **Validating that the final output is comprehensive** and balanced.
All criticism must be constructive. Your goal is to fortify the
research, not invalidate it. Structure your feedback clearly, drawing
attention to specific points for revision. Your overarching aim is to
ensure the final research product meets the highest possible quality
standards.
"""
```

비평 에이전트는 역할, 책임 및 피드백 접근 방식을 개략적으로 설명하는 사전 정의된 시스템 프롬프트(System Prompt)를 기반으로 운영됩니다. 이 에이전트를 위한 잘 설계된 프롬프트는 평가자로서의 기능을 명확하게 확립해야 합니다. 비판적 초점 영역을 명시하고 단순한 거부보다 건설적인 피드백 제공을 강조해야 합니다. 또한 강점과 약점 모두의 식별을 장려하고, 피드백을 구성하고 제시하는 방법에 대해 에이전트를 안내해야 합니다.

## OpenAI를 활용한 실습 코드

이 시스템은 자원 인식 최적화 전략을 사용하여 사용자 쿼리를 효율적으로 처리합니다. 먼저 각 쿼리를 세 가지 범주 중 하나로 분류하여 가장 적절하고 비용 효율적인 처리 경로를 결정합니다. 이 접근 방식은 단순한 요청에 연산 자원을 낭비하지 않으면서도 복잡한 쿼리가 필요한 주의를 받을 수 있도록 합니다. 세 가지 범주는 다음과 같습니다.

- **simple:** 복잡한 추론이나 외부 데이터 없이 직접 답변할 수 있는 간단한 질문.
- **reasoning:** 논리적 추론이나 다단계 사고 과정이 필요한 쿼리로, 더 강력한 모델로 라우팅됩니다.
- **internet_search:** 현재 정보가 필요한 질문으로, 자동으로 Google 검색을 실행하여 최신 답변을 제공합니다.

코드는 MIT 라이선스 하에 있으며 Github에서 이용 가능합니다: (https://github.com/mahtabsyed/21-Agentic-Patterns/blob/main/16_Resource_Aware_Opt_LLM_Reflection_v2.ipynb)

```python
# MIT License
# Copyright (c) 2025 Mahtab Syed
# https://www.linkedin.com/in/mahtabsyed/

import os
import requests
import json
from dotenv import load_dotenv
from openai import OpenAI

# Load environment variables
load_dotenv()
OPENAI_API_KEY = os.getenv("OPENAI_API_KEY")
GOOGLE_CUSTOM_SEARCH_API_KEY =
os.getenv("GOOGLE_CUSTOM_SEARCH_API_KEY")
GOOGLE_CSE_ID = os.getenv("GOOGLE_CSE_ID")

if not OPENAI_API_KEY or not GOOGLE_CUSTOM_SEARCH_API_KEY or not
GOOGLE_CSE_ID:
   raise ValueError(
       "Please set OPENAI_API_KEY, GOOGLE_CUSTOM_SEARCH_API_KEY, and
GOOGLE_CSE_ID in your .env file."
   )

client = OpenAI(api_key=OPENAI_API_KEY)

# --- Step 1: Classify the Prompt ---
def classify_prompt(prompt: str) -> dict:
   system_message = {
       "role": "system",
       "content": (
           "You are a classifier that analyzes user prompts and
returns one of three categories ONLY:\n\n"
           "- simple\n"
           "- reasoning\n"
           "- internet_search\n\n"
           "Rules:\n"
           "- Use 'simple' for direct factual questions that need no
reasoning or current events.\n"
           "- Use 'reasoning' for logic, math, or multi-step
inference questions.\n"
           "- Use 'internet_search' if the prompt refers to current
events, recent data, or things not in your training data.\n\n"
           "Respond ONLY with JSON like:\n"
           '{ "classification": "simple" }'
       ),
   }

   user_message = {"role": "user", "content": prompt}

   response = client.chat.completions.create(
       model="gpt-4o", messages=[system_message, user_message],
temperature=1
   )

   reply = response.choices[0].message.content
   return json.loads(reply)

# --- Step 2: Google Search ---
def google_search(query: str, num_results=1) -> list:
   url = "https://www.googleapis.com/customsearch/v1"
   params = {
       "key": GOOGLE_CUSTOM_SEARCH_API_KEY,
       "cx": GOOGLE_CSE_ID,
       "q": query,
       "num": num_results,
   }

   try:
       response = requests.get(url, params=params)
       response.raise_for_status()
       results = response.json()


       if "items" in results and results["items"]:
           return [
               {
                   "title": item.get("title"),
                   "snippet": item.get("snippet"),
                   "link": item.get("link"),
               }
               for item in results["items"]
           ]
       else:
           return []
   except requests.exceptions.RequestException as e:

def generate_response(prompt: str, classification: str,
search_results=None) -> str:
   if classification == "simple":
       model = "gpt-4o-mini"
       full_prompt = prompt
   elif classification == "reasoning":
       model = "o4-mini"
       full_prompt = prompt
   elif classification == "internet_search":
       model = "gpt-4o"
       # Convert each search result dict to a readable string
       if search_results:
           search_context = "\n".join(
               [
                   f"Title: {item.get('title')}\nSnippet:
{item.get('snippet')}\nLink: {item.get('link')}"
                   for item in search_results

       else:
           search_context = "No search results found."

   response = client.chat.completions.create(
       model=model,

       search_results = google_search(prompt)
       # print("\n🔍 Search Results:", search_results)

search_results)
   return {"classification": classification, "response": answer,
"model": model}
```

- **순차적 모델 폴백:** 이 메커니즘은 사용자가 계층적 모델 목록을 지정할 수 있도록 하여 운영 중복성을 제공합니다. 시스템은 먼저 시퀀스에서 지정된 기본 모델로 요청을 처리하려고 시도합니다. 서비스 불가, 속도 제한, 콘텐츠 필터링 등 여러 오류 조건으로 인해 기본 모델이 응답하지 못하는 경우, 시스템은 목록에서 다음에 지정된 모델로 자동으로 요청을 재라우팅합니다. 이 프로세스는 목록의 모델이 요청을 성공적으로 실행하거나 목록이 소진될 때까지 계속됩니다. 작업의 최종 비용과 응답에 반환된 모델 식별자는 계산을 성공적으로 완료한 모델에 해당합니다.

## 동적 모델 전환을 넘어서: 에이전트 자원 최적화의 스펙트럼

리소스 인식 최적화는 실세계 제약 내에서 효율적이고 효과적으로 운영되는 지능형 에이전트 시스템 개발에 있어 매우 중요합니다. 몇 가지 추가 기법을 살펴보겠습니다.

**동적 모델 전환(Dynamic Model Switching)** 은 당면 작업의 복잡성과 가용 연산 자원을 기반으로 대규모 언어 모델(LLM)을 전략적으로 선택하는 중요한 기법입니다. 단순 쿼리에는 가볍고 비용 효율적인 LLM을 배포하고, 복잡하고 다면적인 문제에는 더 정교하고 자원 집약적인 모델을 활용합니다.

**적응형 툴 사용 및 선택(Adaptive Tool Use & Selection)** 은 에이전트가 도구 모음에서 지능적으로 선택할 수 있도록 하여, API 사용 비용·지연 시간·실행 시간과 같은 요소를 신중하게 고려하면서 각 특정 하위 작업에 가장 적절하고 효율적인 도구를 선택합니다. 이 동적 툴 선택은 외부 API 및 서비스의 사용을 최적화하여 전체 시스템 효율성을 향상시킵니다.

**컨텍스트 정리 및 요약(Contextual Pruning & Summarization)** 은 에이전트가 처리하는 정보량을 관리하는 데 중요한 역할을 하며, 상호작용 기록에서 가장 관련성 높은 정보만을 지능적으로 요약하고 선택적으로 유지함으로써 프롬프트 토큰 수를 전략적으로 최소화하고 추론 비용을 줄이며 불필요한 연산 부하를 방지합니다.

**사전 자원 예측(Proactive Resource Prediction)** 은 미래 워크로드와 시스템 요구사항을 예측함으로써 자원 수요를 예측하며, 이를 통해 자원의 사전 할당 및 관리가 가능하여 시스템 반응성을 보장하고 병목 현상을 방지합니다.

**멀티 에이전트 시스템에서의 비용 민감 탐색(Cost-Sensitive Exploration)** 은 전통적인 연산 비용 외에 통신 비용까지 최적화 고려사항을 확장하여, 에이전트가 협업하고 정보를 공유하는 전략에 영향을 미치며 전체 자원 지출을 최소화하는 것을 목표로 합니다.

**에너지 효율적 배포(Energy-Efficient Deployment)** 는 엄격한 자원 제약이 있는 환경에 특화되어 지능형 에이전트 시스템의 에너지 발자국을 최소화하고, 운영 시간을 연장하며 전체 운영 비용을 절감합니다.

**병렬화 및 분산 컴퓨팅 인식(Parallelization & Distributed Computing Awareness)** 은 분산 자원을 활용하여 에이전트의 처리 능력과 처리량을 향상시키고, 연산 워크로드를 여러 시스템이나 프로세서에 분산하여 더 큰 효율성과 빠른 작업 완료를 달성합니다.

**학습된 자원 배분 정책(Learned Resource Allocation Policies)** 은 학습 메커니즘을 도입하여 에이전트가 피드백과 성능 지표를 기반으로 자원 배분 전략을 시간이 지남에 따라 적응하고 최적화하며, 지속적인 개선을 통해 효율성을 높입니다.

**우아한 성능 저하 및 폴백 메커니즘(Graceful Degradation and Fallback Mechanisms)** 은 자원 제약이 심각한 경우에도 지능형 에이전트 시스템이 아마도 감소된 용량으로나마 계속 기능할 수 있도록 보장하며, 성능을 우아하게 저하시키고 대안 전략으로 폴백하여 운영을 유지하고 필수 기능을 제공합니다.

## 한눈에 보기

**무엇이 문제인가:** 리소스 인식 최적화(Resource-Aware Optimization)는 지능형 시스템에서 연산·시간·비용 자원 소비를 관리하는 과제를 해결합니다. LLM 기반 애플리케이션은 비용이 많이 들고 느릴 수 있으며, 모든 작업에 최적의 모델이나 도구를 선택하는 것은 종종 비효율적입니다. 이는 시스템 출력 품질과 이를 생성하는 데 필요한 자원 간의 근본적인 트레이드오프를 만듭니다. 동적 관리 전략 없이는 시스템이 다양한 작업 복잡도에 적응하거나 예산 및 성능 제약 내에서 운영할 수 없습니다.

**왜 이 패턴인가:** 표준화된 솔루션은 당면 작업에 따라 자원을 지능적으로 모니터링하고 할당하는 에이전틱 시스템을 구축하는 것입니다. 이 패턴은 일반적으로 "라우터 에이전트(Router Agent)"를 사용하여 먼저 들어오는 요청의 복잡성을 분류합니다. 그런 다음 요청은 가장 적합한 LLM이나 도구로 전달됩니다—단순 쿼리에는 빠르고 저렴한 모델, 복잡한 추론에는 더 강력한 모델. "비평 에이전트(Critique Agent)"는 응답 품질을 평가하고 시간이 지남에 따라 라우팅 로직을 개선하기 위한 피드백을 제공함으로써 프로세스를 더욱 정교하게 할 수 있습니다. 이 동적 멀티 에이전트 접근 방식은 시스템이 효율적으로 운영되어 응답 품질과 비용 효율성의 균형을 맞추도록 보장합니다.

**경험 법칙:** API 호출이나 연산 능력에 대한 엄격한 재정 예산 하에서 운영할 때, 빠른 응답 시간이 중요한 지연 시간 민감 애플리케이션을 구축할 때, 제한된 배터리 수명을 가진 엣지 디바이스와 같은 자원 제약 하드웨어에 에이전트를 배포할 때, 응답 품질과 운영 비용 간의 트레이드오프를 프로그래밍 방식으로 균형 잡을 때, 그리고 서로 다른 자원 요구사항을 가진 작업이 있는 복잡한 다단계 워크플로를 관리할 때 이 패턴을 사용하십시오.

## 핵심 정리

- **리소스 인식 최적화의 필수성:** 지능형 에이전트는 연산·시간·비용 자원을 동적으로 관리할 수 있습니다. 모델 사용 및 실행 경로에 관한 결정은 실시간 제약과 목표에 따라 이루어집니다.
- **확장성을 위한 멀티 에이전트 아키텍처:** Google의 에이전트 개발 키트(ADK)는 멀티 에이전트 프레임워크를 제공하여 모듈형 설계를 가능하게 합니다. 서로 다른 에이전트(응답, 라우팅, 비평)가 특정 작업을 처리합니다.
- **동적 LLM 기반 라우팅:** 라우터 에이전트(Router Agent)는 쿼리 복잡도와 예산에 따라 쿼리를 언어 모델(단순 쿼리에는 Gemini Flash, 복잡 쿼리에는 Gemini Pro)로 안내합니다. 이는 비용과 성능을 최적화합니다.
- **비평 에이전트의 기능:** 전용 비평 에이전트(Critique Agent)는 자기 수정(Self-Correction), 성능 모니터링, 라우팅 로직 개선을 위한 피드백을 제공하여 시스템 효과성을 향상시킵니다.
- **피드백과 유연성을 통한 최적화:** 비평 및 모델 통합 유연성을 위한 평가 기능은 적응적이고 자기 개선하는 시스템 동작에 기여합니다.
- **추가 리소스 인식 최적화:** 기타 방법으로는 적응형 툴 사용 및 선택(Adaptive Tool Use & Selection), 컨텍스트 정리 및 요약(Contextual Pruning & Summarization), 사전 자원 예측(Proactive Resource Prediction), 멀티 에이전트 시스템에서의 비용 민감 탐색(Cost-Sensitive Exploration), 에너지 효율적 배포(Energy-Efficient Deployment), 병렬화 및 분산 컴퓨팅 인식(Parallelization & Distributed Computing Awareness), 학습된 자원 배분 정책(Learned Resource Allocation Policies), 우아한 성능 저하 및 폴백 메커니즘(Graceful Degradation and Fallback Mechanisms), 중요 작업의 우선순위화(Prioritization of Critical Tasks)가 있습니다.

## 결론

리소스 인식 최적화는 지능형 에이전트 개발에 필수적으로, 실세계 제약 내에서의 효율적인 운영을 가능하게 합니다. 연산·시간·비용 자원을 관리함으로써 에이전트는 최적의 성능과 비용 효율성을 달성할 수 있습니다. 동적 모델 전환(Dynamic Model Switching), 적응형 툴 사용(Adaptive Tool Use), 컨텍스트 정리(Contextual Pruning)와 같은 기법은 이러한 효율성 달성에 중요합니다. 학습된 자원 배분 정책(Learned Resource Allocation Policies)과 우아한 성능 저하(Graceful Degradation)를 포함한 고급 전략은 다양한 조건에서 에이전트의 적응성과 복원력을 향상시킵니다. 이러한 최적화 원칙을 에이전트 설계에 통합하는 것은 확장 가능하고 견고하며 지속 가능한 AI 시스템을 구축하는 데 기본적입니다.
