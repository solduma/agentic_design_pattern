# 4장: 반성적 검토

## 반성적 검토 패턴 개요

앞선 장들에서 우리는 에이전틱 설계의 기본 패턴들을 살펴보았습니다. 순차 실행을 위한 프롬프트 체이닝(Prompt Chaining), 동적 경로 선택을 위한 라우팅(Routing), 그리고 동시 작업 실행을 위한 병렬화(Parallelization)가 그것입니다. 이러한 패턴들은 에이전트가 복잡한 작업을 더 효율적이고 유연하게 수행할 수 있도록 합니다. 그러나 아무리 정교한 워크플로를 갖추더라도, 에이전트의 초기 출력이나 계획이 최적이거나 정확하거나 완전하지 않을 수 있습니다. 바로 이 지점에서 반성적 검토(Reflection) 패턴이 필요합니다.

반성적 검토(Reflection) 패턴은 에이전트가 자신의 작업, 출력, 또는 내부 상태를 평가하고, 그 평가를 바탕으로 성능을 개선하거나 응답을 정제하는 과정을 의미합니다. 이는 자기 수정(Self-Correction) 또는 자기 개선의 한 형태로, 에이전트가 피드백, 내부 비평, 또는 원하는 기준과의 비교를 통해 출력을 반복적으로 정제하거나 접근 방식을 조정할 수 있게 합니다. 반성적 검토는 때로 초기 에이전트의 출력을 분석하는 역할을 전담하는 별도의 에이전트에 의해 수행되기도 합니다.

단순한 순차 체인처럼 출력이 다음 단계로 직접 전달되거나, 라우팅처럼 경로를 선택하는 것과 달리, 반성적 검토는 피드백 루프(feedback loop)를 도입합니다. 에이전트는 단순히 출력을 생성하는 데 그치지 않고, 그 출력(또는 이를 생성한 과정)을 검토하여 잠재적 문제나 개선 영역을 파악하고, 이를 바탕으로 더 나은 버전을 생성하거나 향후 행동을 수정합니다.

이 과정은 일반적으로 다음 단계로 이루어집니다.

1. **실행(Execution)**: 에이전트가 작업을 수행하거나 초기 출력을 생성합니다.
2. **평가/비평(Evaluation/Critique)**: 에이전트(종종 별도의 LLM 호출 또는 일련의 규칙을 사용)가 이전 단계의 결과를 분석합니다. 이 평가는 사실적 정확성, 일관성, 문체, 완전성, 지시 준수 여부 또는 기타 관련 기준을 확인할 수 있습니다.
3. **반성/정제(Reflection/Refinement)**: 비평을 바탕으로 에이전트가 개선 방향을 결정합니다. 이는 정제된 출력 생성, 후속 단계의 파라미터 조정, 또는 전체 계획 수정 등을 포함할 수 있습니다.
4. **반복(Iteration, 선택적이지만 일반적)**: 정제된 출력이나 조정된 접근 방식을 다시 실행하고, 만족스러운 결과를 얻거나 중단 조건이 충족될 때까지 반성 과정을 반복할 수 있습니다.

반성적 검토 패턴의 핵심적이고 매우 효과적인 구현 방식은 과정을 두 가지 별개의 논리적 역할로 분리하는 것입니다. 바로 **생산자(Producer)** 와 **비평가(Critic)** 입니다. 이를 흔히 "생성자-비평가(Generator-Critic)" 또는 "생산자-검토자(Producer-Reviewer)" 모델이라고 부릅니다. 단일 에이전트도 자기 반성을 수행할 수 있지만, 두 개의 전문화된 에이전트(또는 서로 다른 시스템 프롬프트를 가진 두 번의 별도 LLM 호출)를 사용하면 더 견고하고 편향 없는 결과를 얻을 수 있습니다.

1. **생산자 에이전트(Producer Agent)**: 이 에이전트의 주된 책임은 작업의 초기 실행입니다. 코드 작성, 블로그 게시물 초안 작성, 계획 수립 등 콘텐츠를 생성하는 데 전적으로 집중합니다. 초기 프롬프트를 받아 출력의 첫 번째 버전을 생성합니다.
2. **비평가 에이전트(Critic Agent)**: 이 에이전트의 유일한 목적은 생산자가 생성한 출력을 평가하는 것입니다. 별도의 지시사항이 주어지며, 종종 특정 페르소나(예: "당신은 시니어 소프트웨어 엔지니어입니다", "당신은 꼼꼼한 팩트체커입니다")가 부여됩니다. 비평가의 지시사항은 사실적 정확성, 코드 품질, 문체 요구사항 또는 완전성 등 특정 기준에 따라 생산자의 작업을 분석하도록 안내합니다. 결함을 발견하고, 개선 사항을 제안하며, 구조화된 피드백을 제공하도록 설계됩니다.

이러한 역할 분리는 에이전트가 자신의 작업을 검토할 때 발생하는 "인지적 편향"을 방지하기 때문에 강력합니다. 비평가 에이전트는 오류와 개선 영역을 찾는 데에만 전적으로 집중하며 새로운 시각으로 출력에 접근합니다. 비평가의 피드백은 다시 생산자 에이전트에게 전달되고, 생산자는 이를 바탕으로 새롭고 정제된 출력을 생성합니다. 제공된 LangChain과 에이전트 개발 키트(ADK) 코드 예제는 모두 이 두 에이전트 모델을 구현합니다. LangChain 예제는 특정 "reflector\_prompt"를 사용하여 비평가 페르소나를 생성하는 반면, ADK 예제는 생산자와 검토자 에이전트를 명시적으로 정의합니다.

반성적 검토의 구현에는 이러한 피드백 루프를 포함하는 에이전트 워크플로 구조화가 필요합니다. 이는 코드의 반복 루프나 평가 결과에 따른 상태 관리 및 조건부 전환을 지원하는 프레임워크를 통해 구현할 수 있습니다. 단일 평가 및 정제 단계는 LangChain/LangGraph, ADK, 또는 Crew.AI 체인 내에서 구현할 수 있지만, 진정한 반복적 반성은 일반적으로 더 복잡한 오케스트레이션(Orchestration)을 필요로 합니다.

반성적 검토 패턴은 에이전트가 고품질 출력을 생성하고, 미묘한 뉘앙스의 작업을 처리하며, 어느 정도의 자기 인식과 적응력을 발휘할 수 있도록 하는 데 핵심적입니다. 이는 에이전트를 단순히 지시를 수행하는 것에서 더 정교한 문제 해결 및 콘텐츠 생성 방식으로 나아가게 합니다.

목표 설정 및 모니터링(11장 참조)과 반성적 검토의 교차점에도 주목할 필요가 있습니다. 목표는 에이전트의 자기 평가를 위한 궁극적인 기준을 제공하고, 모니터링은 진행 상황을 추적합니다. 많은 실제 사례에서, 반성적 검토는 수정 엔진으로서 모니터링된 피드백을 사용하여 이탈을 분석하고 전략을 조정하는 역할을 합니다. 이러한 시너지는 에이전트를 수동적인 실행자에서 목표를 달성하기 위해 적응적으로 작동하는 목적 지향 시스템으로 변환합니다.

또한 LLM이 대화 기록을 메모리 관리(Memory Management)(8장 참조)를 통해 유지할 때 반성적 검토 패턴의 효과가 크게 향상됩니다. 이 대화 기록은 평가 단계에 중요한 맥락을 제공하여, 에이전트가 출력을 고립된 상태가 아닌 이전 상호작용, 사용자 피드백, 발전하는 목표의 배경 속에서 평가할 수 있게 합니다. 과거의 비평에서 배우고 같은 오류를 반복하지 않도록 합니다. 메모리 없이는 각 반성이 독립적인 사건이지만, 메모리와 함께라면 반성은 각 사이클이 이전 것을 기반으로 쌓이는 누적 과정이 되어 더 지능적이고 맥락 인식적인 정제로 이어집니다.

## 실용적 응용 및 사용 사례

반성적 검토 패턴은 출력 품질, 정확성, 또는 복잡한 제약 조건 준수가 중요한 시나리오에서 가치를 발휘합니다.

### 1. 창의적 글쓰기 및 콘텐츠 생성

생성된 텍스트, 이야기, 시, 마케팅 카피 등을 정제합니다.

- **사용 사례**: 블로그 게시물을 작성하는 에이전트
  - **반성적 검토**: 초안을 생성하고, 흐름, 어조, 명확성에 대해 비평한 다음, 비평을 바탕으로 재작성합니다. 게시물이 품질 기준을 충족할 때까지 반복합니다.
  - **이점**: 더 세련되고 효과적인 콘텐츠를 생산합니다.

### 2. 코드 생성(Code Generation) 및 디버깅

코드를 작성하고, 오류를 식별하며, 수정합니다.

- **사용 사례**: Python 함수를 작성하는 에이전트
  - **반성적 검토**: 초기 코드를 작성하고, 테스트 또는 정적 분석을 실행하며, 오류나 비효율을 식별한 다음, 결과를 바탕으로 코드를 수정합니다.
  - **이점**: 더 견고하고 기능적인 코드를 생성합니다.

### 3. 복잡한 문제 해결

다단계 추론 작업에서 중간 단계나 제안된 해결책을 평가합니다.

- **사용 사례**: 논리 퍼즐을 푸는 에이전트
  - **반성적 검토**: 단계를 제안하고, 해당 단계가 해결책에 가까이 이르는지 또는 모순을 도입하는지 평가하며, 필요하면 되돌아가거나 다른 단계를 선택합니다.
  - **이점**: 복잡한 문제 공간을 탐색하는 에이전트의 능력을 향상시킵니다.

### 4. 요약 및 정보 합성

정확성, 완전성, 간결성을 위해 요약을 정제합니다.

- **사용 사례**: 긴 문서를 요약하는 에이전트
  - **반성적 검토**: 초기 요약을 생성하고, 원본 문서의 핵심 내용과 비교하며, 누락된 정보를 포함하거나 정확성을 높이기 위해 요약을 정제합니다.
  - **이점**: 더 정확하고 포괄적인 요약을 생성합니다.

### 5. 계획 수립과 전략

제안된 계획을 평가하고 잠재적 결함이나 개선 사항을 식별합니다.

- **사용 사례**: 목표를 달성하기 위한 일련의 행동을 계획하는 에이전트
  - **반성적 검토**: 계획을 생성하고, 실행을 시뮬레이션하거나 제약 조건에 대한 실현 가능성을 평가하며, 평가를 바탕으로 계획을 수정합니다.
  - **이점**: 더 효과적이고 현실적인 계획을 개발합니다.

### 6. 대화 에이전트

이전 대화 턴을 검토하여 맥락을 유지하고, 오해를 수정하거나 응답 품질을 향상시킵니다.

- **사용 사례**: 고객 지원 챗봇
  - **반성적 검토**: 사용자 응답 후, 대화 기록과 마지막으로 생성된 메시지를 검토하여 일관성을 확인하고 사용자의 최신 입력을 정확하게 다루었는지 확인합니다.
  - **이점**: 더 자연스럽고 효과적인 대화로 이어집니다.

반성적 검토는 에이전틱 시스템에 메타인지의 계층을 추가하여, 자신의 출력과 과정에서 배울 수 있게 함으로써 더 지능적이고, 신뢰할 수 있으며, 고품질의 결과를 이끌어냅니다.

## 실습 코드 예제 (LangChain)

완전한 반복적 반성 과정의 구현에는 상태 관리와 순환 실행을 위한 메커니즘이 필요합니다. LangGraph와 같은 그래프 기반 프레임워크나 커스텀 절차적 코드에서는 이를 기본으로 지원하지만, 단일 반성 사이클의 기본 원리는 LCEL(LangChain Expression Language)의 조합 문법을 사용하여 효과적으로 시연할 수 있습니다.

이 예제는 LangChain 라이브러리와 OpenAI의 GPT-4o 모델을 사용하여 반성 루프를 구현합니다. 숫자의 팩토리얼을 계산하는 Python 함수를 반복적으로 생성하고 정제합니다. 프로세스는 작업 프롬프트로 시작하여 초기 코드를 생성하고, 시니어 소프트웨어 엔지니어 역할을 수행하는 비평가의 비평을 바탕으로 각 반복에서 코드를 정제하여, 비평 단계에서 코드가 완벽하다고 판단되거나 최대 반복 횟수에 도달할 때까지 계속됩니다. 최종적으로 정제된 코드를 출력합니다.

먼저 필요한 라이브러리를 설치하십시오.

```bash
pip install langchain langchain-community langchain-openai
```

또한 선택한 언어 모델(예: OpenAI, Google Gemini, Anthropic)의 API 키로 환경을 설정해야 합니다.

```python
import os
from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.messages import SystemMessage, HumanMessage

# --- Configuration ---
# Load environment variables from .env file (for OPENAI_API_KEY)
load_dotenv()

# Check if the API key is set
if not os.getenv("OPENAI_API_KEY"):
   raise ValueError("OPENAI_API_KEY not found in .env file. Please
add it.")

# Initialize the Chat LLM. We use gpt-4o for better reasoning.
# A lower temperature is used for more deterministic outputs.
llm = ChatOpenAI(model="gpt-4o", temperature=0.1)

def run_reflection_loop():
   """
   Demonstrates a multi-step AI reflection loop to progressively
improve a Python function.
   """
   # --- The Core Task ---
   task_prompt = """
   Your task is to create a Python function named
`calculate_factorial`.
   This function should do the following:
   1.  Accept a single integer `n` as input.
```

```python
   2.  Calculate its factorial (n!).
   3.  Include a clear docstring explaining what the function does.
   4.  Handle edge cases: The factorial of 0 is 1.
   5.  Handle invalid input: Raise a ValueError if the input is a
negative number.
   """
   # --- The Reflection Loop ---
   max_iterations = 3
   current_code = ""
   # We will build a conversation history to provide context in each
step.
   message_history = [HumanMessage(content=task_prompt)]


   for i in range(max_iterations):
       print("\n" + "="*25 + f" REFLECTION LOOP: ITERATION {i + 1} "
+ "="*25)

       # --- 1. GENERATE / REFINE STAGE ---
       # In the first iteration, it generates. In subsequent
iterations, it refines.
       if i == 0:
           print("\n>>> STAGE 1: GENERATING initial code...")
           # The first message is just the task prompt.
           response = llm.invoke(message_history)
           current_code = response.content
       else:
           print("\n>>> STAGE 1: REFINING code based on previous
critique...")
           # The message history now contains the task,
           # the last code, and the last critique.
           # We instruct the model to apply the critiques.
           message_history.append(HumanMessage(content="Please refine
the code using the critiques provided."))
           response = llm.invoke(message_history)
           current_code = response.content

       print("\n--- Generated Code (v" + str(i + 1) + ") ---\n" +
current_code)
       message_history.append(response) # Add the generated code to
history

       # --- 2. REFLECT STAGE ---
       print("\n>>> STAGE 2: REFLECTING on the generated code...")

       # Create a specific prompt for the reflector agent.
       # This asks the model to act as a senior code reviewer.
```

```python
       reflector_prompt = [
           SystemMessage(content="""
               You are a senior software engineer and an expert
               in Python.
               Your role is to perform a meticulous code review.
               Critically evaluate the provided Python code based
               on the original task requirements.
               Look for bugs, style issues, missing edge cases,
               and areas for improvement.
               If the code is perfect and meets all requirements,
               respond with the single phrase 'CODE_IS_PERFECT'.
               Otherwise, provide a bulleted list of your critiques.
           """),
           HumanMessage(content=f"Original
Task:\n{task_prompt}\n\nCode to Review:\n{current_code}")
       ]

       critique_response = llm.invoke(reflector_prompt)
       critique = critique_response.content

       # --- 3. STOPPING CONDITION ---
       if "CODE_IS_PERFECT" in critique:
           print("\n--- Critique ---\nNo further critiques found. The
code is satisfactory.")
           break

       print("\n--- Critique ---\n" + critique)
       # Add the critique to the history for the next refinement
loop.
       message_history.append(HumanMessage(content=f"Critique of the
previous code:\n{critique}"))

   print("\n" + "="*30 + " FINAL RESULT " + "="*30)
   print("\nFinal refined code after the reflection process:\n")
   print(current_code)

if __name__ == "__main__":
   run_reflection_loop()
```

이 코드는 환경 설정부터 시작합니다. API 키를 로드하고, 집중된 출력을 위해 낮은 temperature로 GPT-4o와 같은 강력한 언어 모델을 초기화합니다. 핵심 작업은 팩토리얼을 계산하는 Python 함수를 작성하도록 요청하는 프롬프트로 정의됩니다. 여기에는 독스트링, 엣지 케이스(0의 팩토리얼), 음수 입력에 대한 오류 처리 등 구체적인 요구사항이 포함됩니다. `run_reflection_loop` 함수가 반복적 정제 과정을 오케스트레이션합니다. 루프 내에서 첫 번째 반복 시 언어 모델이 작업 프롬프트를 바탕으로 초기 코드를 생성합니다. 이후 반복에서는 이전 단계의 비평을 바탕으로 코드를 정제합니다. 별도의 "리플렉터" 역할(언어 모델이 맡지만 다른 시스템 프롬프트 사용)이 시니어 소프트웨어 엔지니어로서 생성된 코드를 원래 작업 요구사항에 따라 비평합니다. 이 비평은 문제의 불릿 목록 또는 문제가 없을 경우 'CODE\_IS\_PERFECT' 문구로 제공됩니다. 비평이 코드가 완벽하다고 나타내거나 최대 반복 횟수에 도달할 때까지 루프가 계속됩니다. 대화 기록은 생성/정제 및 반성 단계 모두에서 맥락을 제공하기 위해 각 단계에서 언어 모델에 전달되며 유지됩니다. 마지막으로 스크립트는 루프가 종료된 후 마지막으로 생성된 코드 버전을 출력합니다.

## 실습 코드 예제 (ADK)

이제 Google ADK를 사용하여 구현한 개념적 코드 예제를 살펴보겠습니다. 이 코드는 생성자-비평가 구조를 활용하여 반성적 검토를 시연합니다. 하나의 컴포넌트(생성자)가 초기 결과나 계획을 생성하고, 다른 컴포넌트(비평가)가 비판적 피드백을 제공하여 생성자를 더 정제되고 정확한 최종 출력으로 안내합니다.

```python
from google.adk.agents import SequentialAgent, LlmAgent

# The first agent generates the initial draft.
generator = LlmAgent(
   name="DraftWriter",
   description="Generates initial draft content on a given subject.",
   instruction="Write a short, informative paragraph about the user's
subject.",
   output_key="draft_text" # The output is saved to this state key.
)

# The second agent critiques the draft from the first agent.
reviewer = LlmAgent(
   name="FactChecker",
   description="Reviews a given text for factual accuracy and
provides a structured critique.",
   instruction="""
   You are a meticulous fact-checker.
   1. Read the text provided in the state key 'draft_text'.
   2. Carefully verify the factual accuracy of all claims.
   3. Your final output must be a dictionary containing two keys:
      - "status": A string, either "ACCURATE" or "INACCURATE".
      - "reasoning": A string providing a clear explanation for your
status, citing specific issues if any are found.
```

```python
   """,
   output_key="review_output" # The structured dictionary is saved
here.
)

# The SequentialAgent ensures the generator runs before the reviewer.
review_pipeline = SequentialAgent(
   name="WriteAndReview_Pipeline",
   sub_agents=[generator, reviewer]
)

# Execution Flow:
# 1. generator runs -> saves its paragraph to state['draft_text'].
# 2. reviewer runs -> reads state['draft_text'] and saves its
dictionary output to state['review_output'].
```

이 코드는 Google ADK에서 텍스트를 생성하고 검토하는 순차 에이전트 파이프라인의 사용을 보여줍니다. 두 개의 `LlmAgent` 인스턴스인 `generator`와 `reviewer`를 정의합니다. `generator` 에이전트는 주어진 주제에 대한 초기 초안 단락을 작성하도록 설계되었으며, 짧고 유익한 내용을 작성하도록 지시받고 출력을 상태 키 `draft_text`에 저장합니다. `reviewer` 에이전트는 생성자가 작성한 텍스트의 팩트체커 역할을 합니다. `draft_text`에서 텍스트를 읽고 사실적 정확성을 검증하도록 지시받습니다. 검토자의 출력은 두 키를 가진 구조화된 딕셔너리입니다. `status`는 텍스트가 "ACCURATE"인지 "INACCURATE"인지를 나타내고, `reasoning`은 상태에 대한 설명을 제공합니다. 이 딕셔너리는 상태 키 `review_output`에 저장됩니다. `review_pipeline`이라는 이름의 `SequentialAgent`가 두 에이전트의 실행 순서를 관리하기 위해 생성됩니다. 이를 통해 `generator`가 먼저 실행되고 이어서 `reviewer`가 실행되도록 보장합니다. 전체 실행 흐름은 `generator`가 텍스트를 생성하여 상태에 저장하고, 이후 `reviewer`가 상태에서 이 텍스트를 읽어 팩트 체킹을 수행한 뒤 결과(`status`와 `reasoning`)를 다시 상태에 저장하는 것입니다. 이 파이프라인은 별도의 에이전트를 사용하여 콘텐츠 생성과 검토의 구조화된 프로세스를 가능하게 합니다. 참고로, ADK의 `LoopAgent`를 활용한 대안적 구현도 관심 있는 분들에게 제공됩니다.

결론을 내리기 전에, 반성적 검토 패턴이 출력 품질을 크게 향상시키는 반면 중요한 트레이드오프가 있다는 점을 고려해야 합니다. 강력하긴 하지만 반복 과정은 더 높은 비용과 지연 시간(Latency)으로 이어질 수 있습니다. 각 정제 루프마다 새로운 LLM 호출이 필요할 수 있어 시간에 민감한 애플리케이션에는 최적이 아닐 수 있습니다. 또한 이 패턴은 메모리 집약적입니다. 각 반복마다 초기 출력, 비평, 후속 정제를 포함하는 대화 기록이 확장됩니다.

## 한눈에 보기

**무엇이 문제인가**: 에이전트의 초기 출력은 부정확성, 불완전성, 또는 복잡한 요구사항 충족 실패로 인해 종종 최적이 아닙니다. 기본적인 에이전틱 워크플로는 에이전트가 자신의 오류를 인식하고 수정하기 위한 내장 프로세스를 갖추지 못합니다. 이는 에이전트가 자신의 작업을 평가하도록 하거나, 더 견고하게는 초기 응답이 품질에 상관없이 최종 응답이 되는 것을 방지하기 위해 별도의 논리적 에이전트를 비평가로 도입함으로써 해결됩니다.

**왜 반성적 검토인가**: 반성적 검토(Reflection) 패턴은 자기 수정(Self-Correction)과 정제를 위한 메커니즘을 도입함으로써 해결책을 제시합니다. "생산자" 에이전트가 출력을 생성하고 "비평가" 에이전트(또는 생산자 자신)가 미리 정의된 기준에 따라 평가하는 피드백 루프를 확립합니다. 이 비평은 개선된 버전을 생성하는 데 사용됩니다. 생성, 평가, 정제의 반복적 과정은 최종 결과의 품질을 점진적으로 향상시켜 더 정확하고, 일관되며, 신뢰할 수 있는 결과로 이어집니다.

**경험 법칙**: 최종 출력의 품질, 정확성, 세부 사항이 속도와 비용보다 더 중요할 때 반성적 검토 패턴을 사용하십시오. 세련된 장문 콘텐츠 생성, 코드 작성 및 디버깅, 상세한 계획 수립 등의 작업에 특히 효과적입니다. 작업에 높은 객관성이나 범용 생산자 에이전트가 놓칠 수 있는 전문화된 평가가 필요할 때 별도의 비평가 에이전트를 사용하십시오.

## 핵심 정리

- 반성적 검토(Reflection) 패턴의 주요 장점은 출력을 반복적으로 자기 수정(Self-Correction)하고 정제하여 품질, 정확성, 복잡한 지시 준수도를 크게 높이는 능력입니다.
- 실행, 평가/비평, 정제의 피드백 루프로 구성됩니다. 반성적 검토는 고품질, 정확, 또는 미묘한 출력이 필요한 작업에 필수적입니다.
- 강력한 구현 방식은 생산자-비평가(Producer-Critic) 모델로, 별도의 에이전트(또는 지시된 역할)가 초기 출력을 평가합니다. 이 역할 분리는 객관성을 높이고 더 전문화되고 구조화된 피드백을 가능하게 합니다.
- 그러나 이러한 이점은 지연 시간(Latency) 증가와 계산 비용, 그리고 모델의 컨텍스트 윈도우 초과 또는 API 서비스 스로틀링의 높은 위험을 수반합니다.
- 완전한 반복 반성은 종종 상태 기반 워크플로(LangGraph 등)를 필요로 하지만, 단일 반성 단계는 LangChain의 LCEL을 사용하여 출력을 비평에 전달하고 이후 정제하는 방식으로 구현할 수 있습니다.
- Google ADK는 한 에이전트의 출력이 다른 에이전트에 의해 비평되는 순차 워크플로를 통해 반성적 검토를 지원하여 후속 정제 단계를 허용합니다.
- 이 패턴은 에이전트가 자기 수정(Self-Correction)을 수행하고 시간이 지남에 따라 성능을 향상시킬 수 있게 합니다.

## 결론

반성적 검토 패턴은 에이전트의 워크플로 내에서 자기 수정(Self-Correction)을 위한 핵심 메커니즘을 제공하여, 단일 실행을 넘어 반복적 개선을 가능하게 합니다. 이는 시스템이 출력을 생성하고, 특정 기준에 따라 평가하며, 그 평가를 사용하여 정제된 결과를 생성하는 루프를 만듦으로써 달성됩니다. 이 평가는 에이전트 자신에 의해(자기 반성) 또는 더 효과적으로는 별개의 비평가 에이전트에 의해 수행될 수 있으며, 이는 패턴 내의 핵심 아키텍처 선택을 나타냅니다.

완전한 자율적 다단계 반성 과정은 상태 관리를 위한 견고한 아키텍처를 필요로 하지만, 핵심 원리는 단일 생성-비평-정제 사이클에서 효과적으로 시연됩니다. 제어 구조로서, 반성적 검토는 다른 기본 패턴들과 통합되어 더 견고하고 기능적으로 복잡한 에이전틱 시스템을 구성할 수 있습니다.
