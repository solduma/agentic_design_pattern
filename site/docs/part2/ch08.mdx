# 8장: 메모리 관리

효과적인 메모리 관리(Memory Management)는 에이전트가 정보를 지속적으로 보유하는 데 필수적입니다. 에이전트는 인간과 마찬가지로 효율적으로 동작하기 위해 다양한 유형의 기억이 필요합니다. 이 장에서는 메모리 관리를 심층적으로 다루며, 에이전트의 즉각적(단기) 기억과 지속적(장기) 기억 요구사항을 구체적으로 살펴봅니다.

에이전트 시스템에서 메모리란 에이전트가 과거 상호작용, 관찰, 학습 경험으로부터 얻은 정보를 보유하고 활용하는 능력을 의미합니다. 이 능력을 통해 에이전트는 충분한 정보에 기반한 의사결정을 내리고, 대화 맥락을 유지하며, 시간이 지남에 따라 성능을 개선할 수 있습니다. 에이전트 메모리는 일반적으로 두 가지 주요 유형으로 분류됩니다.

- **단기 기억(컨텍스트 기억):** 작업 기억과 유사하게, 현재 처리 중이거나 최근에 접근한 정보를 보유합니다. 대규모 언어 모델(LLM)을 사용하는 에이전트의 경우, 단기 기억은 주로 컨텍스트 윈도우 내에 존재합니다. 이 윈도우에는 최근 메시지, 에이전트 응답, 도구 사용 결과, 현재 상호작용에서의 에이전트 반성이 포함되며, 이 모두가 LLM의 후속 응답과 행동에 영향을 미칩니다. 컨텍스트 윈도우는 제한된 용량을 가지고 있어 에이전트가 직접 접근할 수 있는 최근 정보의 양을 제한합니다. 효율적인 단기 기억 관리는 오래된 대화 구간을 요약하거나 핵심 세부사항을 강조하는 방식으로 이 제한된 공간 내에서 가장 관련성 높은 정보를 유지하는 것을 포함합니다. '긴 컨텍스트' 윈도우를 갖는 모델의 등장은 단기 기억의 크기를 단순히 확장할 뿐이며, 단일 상호작용 내에서 더 많은 정보를 보유할 수 있게 합니다. 그러나 이 컨텍스트는 여전히 일시적이며 세션(Session)이 종료되면 사라지고, 매번 처리하는 데 비용이 크고 비효율적일 수 있습니다. 따라서 에이전트는 진정한 지속성을 달성하고, 과거 상호작용에서 정보를 회상하며, 지속적인 지식 기반을 구축하기 위해 별도의 메모리 유형이 필요합니다.

- **장기 기억(지속 기억):** 다양한 상호작용, 작업, 또는 장기간에 걸쳐 에이전트가 보유해야 하는 정보를 위한 저장소 역할을 합니다. 데이터는 일반적으로 에이전트의 즉각적인 처리 환경 외부, 즉 데이터베이스, 지식 그래프, 또는 벡터 데이터베이스에 저장됩니다. 벡터 데이터베이스에서는 정보가 수치 벡터로 변환되어 저장되며, 이를 통해 에이전트는 정확한 키워드 일치보다는 의미론적 유사성을 기반으로 데이터를 검색할 수 있습니다. 이 과정을 시맨틱 검색(semantic search)이라고 합니다. 에이전트가 장기 기억에서 정보가 필요할 때, 외부 저장소를 조회하고 관련 데이터를 검색하여 즉각적인 사용을 위해 단기 컨텍스트에 통합함으로써 이전 지식과 현재 상호작용을 결합합니다.

## 실제 응용 및 활용 사례

메모리 관리는 에이전트가 정보를 추적하고 시간이 지남에 따라 지능적으로 수행하는 데 필수적입니다. 이는 에이전트가 기본적인 질의응답 능력을 초월하는 데 꼭 필요합니다. 응용 사례는 다음과 같습니다.

- **챗봇 및 대화형 AI:** 대화 흐름을 유지하는 것은 단기 기억에 의존합니다. 챗봇은 일관된 응답을 제공하기 위해 이전 사용자 입력을 기억해야 합니다. 장기 기억은 챗봇이 사용자 선호도, 과거 문제, 또는 이전 대화를 회상하여 개인화되고 지속적인 상호작용을 제공할 수 있게 합니다.

- **작업 지향 에이전트:** 다단계 작업을 관리하는 에이전트는 이전 단계, 현재 진행 상황, 전체 목표를 추적하기 위해 단기 기억이 필요합니다. 이 정보는 작업의 컨텍스트나 임시 저장소에 존재할 수 있습니다. 장기 기억은 즉각적인 컨텍스트에 없는 특정 사용자 관련 데이터에 접근하는 데 필수적입니다.

- **개인화된 경험:** 맞춤형 상호작용을 제공하는 에이전트는 장기 기억을 활용하여 사용자 선호도, 과거 행동, 개인 정보를 저장하고 검색합니다. 이를 통해 에이전트는 응답과 제안을 적응시킬 수 있습니다.

- **학습 및 개선:** 에이전트는 과거 상호작용에서 학습하여 성능을 향상시킬 수 있습니다. 성공적인 전략, 실수, 새로운 정보가 장기 기억에 저장되어 미래의 적응을 촉진합니다. 강화학습 에이전트는 이런 방식으로 학습된 전략이나 지식을 저장합니다.

- **정보 검색(RAG):** 질문에 답하도록 설계된 에이전트는 종종 검색 증강 생성(RAG, Retrieval Augmented Generation) 내에서 구현되는 지식 기반, 즉 장기 기억에 접근합니다. 에이전트는 관련 문서나 데이터를 검색하여 응답에 활용합니다.

- **자율 시스템:** 로봇이나 자율주행 자동차는 지도, 경로, 물체 위치, 학습된 행동을 위한 메모리가 필요합니다. 여기에는 즉각적인 주변 환경을 위한 단기 기억과 일반적인 환경 지식을 위한 장기 기억이 포함됩니다.

메모리는 에이전트가 이력을 유지하고, 학습하며, 상호작용을 개인화하고, 복잡한 시간 의존적 문제를 관리할 수 있게 합니다.

## 실습 코드: Google ADK에서의 메모리 관리

Google Agent Developer Kit(ADK)는 컨텍스트와 메모리를 관리하기 위한 구조화된 방법을 제공하며, 실제 응용을 위한 구성요소를 포함합니다. ADK의 세션(Session), 상태(State), 메모리에 대한 확실한 이해는 정보를 보유해야 하는 에이전트를 구축하는 데 필수적입니다.

인간의 상호작용과 마찬가지로, 에이전트는 일관성 있고 자연스러운 대화를 위해 이전 교환을 회상할 수 있어야 합니다. ADK는 세 가지 핵심 개념과 그와 관련된 서비스를 통해 컨텍스트 관리를 단순화합니다.

에이전트와의 모든 상호작용은 고유한 대화 스레드로 간주될 수 있습니다. 에이전트는 이전 상호작용의 데이터에 접근해야 할 수도 있습니다. ADK는 이를 다음과 같이 구조화합니다.

- **세션(Session):** 해당 특정 상호작용에 대한 메시지와 행동(이벤트)을 기록하고, 그 대화와 관련된 임시 데이터(상태)도 저장하는 개별 채팅 스레드입니다.
- **상태(State, session.state):** 세션(Session) 내에 저장된 데이터로, 현재 활성화된 채팅 스레드에만 관련된 정보를 포함합니다.
- **메모리:** 다양한 과거 채팅이나 외부 소스에서 가져온 정보의 검색 가능한 저장소로, 현재 대화를 넘어선 데이터 검색을 위한 리소스 역할을 합니다.

ADK는 복잡하고 상태를 유지하며 맥락을 인식하는 에이전트를 구축하는 데 필수적인 핵심 구성요소를 관리하기 위한 전용 서비스를 제공합니다. SessionService는 채팅 스레드(Session 객체)를 시작, 기록, 종료를 처리하여 관리하며, MemoryService는 장기 지식(Memory)의 저장과 검색을 감독합니다.

SessionService와 MemoryService 모두 다양한 구성 옵션을 제공하여 애플리케이션 요구에 따라 저장 방법을 선택할 수 있습니다. 테스트 목적으로는 인메모리 옵션이 제공되지만, 재시작 시 데이터가 유지되지 않습니다. 영속적인 저장과 확장성을 위해 ADK는 데이터베이스 및 클라우드 기반 서비스도 지원합니다.

## 세션(Session): 각 채팅 추적

ADK의 Session 객체는 개별 채팅 스레드를 추적하고 관리하도록 설계되었습니다. 에이전트와의 대화가 시작될 때, SessionService는 `google.adk.sessions.Session`으로 표현되는 Session 객체를 생성합니다. 이 객체는 고유 식별자(id, app_name, user_id), Event 객체로서의 이벤트 시간순 기록, 세션 특화 임시 데이터 저장 영역인 상태(State), 마지막 업데이트 타임스탬프(last_update_time)를 포함한 특정 대화 스레드와 관련된 모든 데이터를 캡슐화합니다. 개발자는 일반적으로 SessionService를 통해 간접적으로 Session 객체와 상호작용합니다. SessionService는 새 세션 시작, 이전 세션 재개, 세션 활동 기록(상태 업데이트 포함), 활성 세션 식별, 세션 데이터 제거 관리를 포함한 대화 세션의 생명주기 관리를 담당합니다. ADK는 세션 이력과 임시 데이터를 위한 다양한 저장 메커니즘을 가진 여러 SessionService 구현을 제공합니다. 예를 들어, InMemorySessionService는 테스트에 적합하지만 애플리케이션 재시작 시 데이터 지속성을 제공하지 않습니다.

````python
# Example: Using InMemorySessionService
# This is suitable for local development and testing where data
# persistence across application restarts is not required.
from google.adk.sessions import InMemorySessionService
session_service = InMemorySessionService()
````

데이터베이스에 안정적으로 저장하려면 DatabaseSessionService를 사용할 수 있습니다.

````python
# Example: Using DatabaseSessionService
# This is suitable for production or development requiring persistent
storage.
# You need to configure a database URL (e.g., for SQLite, PostgreSQL,
etc.).
# Requires: pip install google-adk[sqlalchemy] and a database driver
(e.g., psycopg2 for PostgreSQL)
from google.adk.sessions import DatabaseSessionService
# Example using a local SQLite file:
db_url = "sqlite:///./my_agent_data.db"
session_service = DatabaseSessionService(db_url=db_url)
````

Google Cloud의 확장 가능한 프로덕션 환경을 위해서는 Vertex AI 인프라를 사용하는 VertexAiSessionService가 있습니다.

````python
# Example: Using VertexAiSessionService
# This is suitable for scalable production on Google Cloud Platform,
leveraging
# Vertex AI infrastructure for session management.
# Requires: pip install google-adk[vertexai] and GCP
setup/authentication
from google.adk.sessions import VertexAiSessionService

PROJECT_ID = "your-gcp-project-id" # Replace with your GCP project ID
LOCATION = "us-central1" # Replace with your desired GCP location
# The app_name used with this service should correspond to the
Reasoning Engine ID or name
REASONING_ENGINE_APP_NAME =
"projects/your-gcp-project-id/locations/us-central1/reasoningEngines/
your-engine-id" # Replace with your Reasoning Engine resource name

session_service = VertexAiSessionService(project=PROJECT_ID,
location=LOCATION)
# When using this service, pass REASONING_ENGINE_APP_NAME to service
methods:
# session_service.create_session(app_name=REASONING_ENGINE_APP_NAME,
...)
# session_service.get_session(app_name=REASONING_ENGINE_APP_NAME,
...)
# session_service.append_event(session, event,
app_name=REASONING_ENGINE_APP_NAME)
# session_service.delete_session(app_name=REASONING_ENGINE_APP_NAME,
...)
````

적절한 SessionService를 선택하는 것은 에이전트의 상호작용 이력과 임시 데이터가 어떻게 저장되고 지속되는지를 결정하므로 매우 중요합니다.

각 메시지 교환은 순환 프로세스를 포함합니다. 메시지가 수신되면, Runner는 SessionService를 사용하여 세션을 검색하거나 새로 생성하고, 에이전트는 세션의 컨텍스트(상태와 이력)를 사용하여 메시지를 처리하며, 에이전트는 응답을 생성하고 상태를 업데이트할 수 있고, Runner는 이를 Event로 캡슐화하며, `session_service.append_event` 메서드가 새 이벤트를 기록하고 저장소의 상태를 업데이트합니다. 그러면 세션은 다음 메시지를 기다립니다. 이상적으로는 상호작용이 종료될 때 `delete_session` 메서드를 사용하여 세션을 종료합니다. 이 프로세스는 SessionService가 세션 특화 이력과 임시 데이터를 관리하여 연속성을 어떻게 유지하는지를 보여줍니다.

## 상태(State): 세션의 스크래치패드

ADK에서 채팅 스레드를 나타내는 각 세션(Session)은 해당 특정 대화 기간 동안 에이전트의 임시 작업 메모리와 유사한 상태(State) 구성요소를 포함합니다. `session.events`가 전체 채팅 이력을 기록하는 반면, `session.state`는 활성 채팅과 관련된 동적 데이터 포인트를 저장하고 업데이트합니다.

근본적으로 `session.state`는 딕셔너리로 작동하며, 키-값 쌍으로 데이터를 저장합니다. 그 핵심 기능은 에이전트가 사용자 선호도, 작업 진행 상황, 점진적 데이터 수집, 또는 후속 에이전트 행동에 영향을 미치는 조건 플래그와 같이 일관된 대화에 필수적인 세부사항을 보유하고 관리할 수 있게 하는 것입니다.

상태의 구조는 문자열 키와 문자열, 숫자, 불리언, 목록, 이러한 기본 유형을 포함하는 딕셔너리와 같은 직렬화 가능한 Python 유형의 값으로 구성됩니다. 상태는 동적이며 대화 전반에 걸쳐 진화합니다. 이러한 변경의 영속성은 구성된 SessionService에 따라 달라집니다.

상태 조직은 데이터 범위와 영속성을 정의하기 위해 키 접두사를 사용하여 달성할 수 있습니다. 접두사 없는 키는 세션 특화입니다.

- `user:` 접두사는 모든 세션에 걸쳐 사용자 ID와 데이터를 연결합니다.
- `app:` 접두사는 애플리케이션의 모든 사용자 간에 공유되는 데이터를 지정합니다.
- `temp:` 접두사는 현재 처리 턴에만 유효하며 영속적으로 저장되지 않는 데이터를 나타냅니다.

에이전트는 단일 `session.state` 딕셔너리를 통해 모든 상태 데이터에 접근합니다. SessionService는 데이터 검색, 병합, 영속성을 처리합니다. 상태는 `session_service.append_event()`를 통해 세션 이력에 Event를 추가할 때 업데이트되어야 합니다. 이는 정확한 추적, 영속 서비스에서의 적절한 저장, 상태 변경의 안전한 처리를 보장합니다.

**1. 간단한 방법: output_key 사용 (에이전트 텍스트 응답용):** 에이전트의 최종 텍스트 응답을 상태에 직접 저장하려면 이 방법이 가장 쉽습니다. LlmAgent를 설정할 때 사용하려는 `output_key`를 지정하면 됩니다. Runner는 이를 확인하고 이벤트를 추가할 때 응답을 상태에 저장하는 데 필요한 액션을 자동으로 생성합니다. `output_key`를 통한 상태 업데이트를 보여주는 코드 예제를 살펴보겠습니다.

````python
# Import necessary classes from the Google Agent Developer Kit
(ADK)

from google.adk.runners import Runner
from google.genai.types import Content, Part

print(f"Initial state: {session.state}")

# --- Run the Agent ---
user_message = Content(parts=[Part(text="Hello")])
print("\n--- Running the agent ---")
for event in runner.run(
   user_id=user_id,
   session_id=session_id,
   new_message=user_message
):
   if event.is_final_response():
     print("Agent responded.")

# --- Check Updated State ---
# Correctly check the state *after* the runner has finished
````

내부적으로 Runner는 `output_key`를 확인하고 `append_event`를 호출할 때 `state_delta`가 포함된 필요한 액션을 자동으로 생성합니다.

**2. 표준 방법: EventActions.state_delta 사용 (더 복잡한 업데이트용):** 여러 키를 한 번에 업데이트하거나, 텍스트가 아닌 항목을 저장하거나, `user:` 또는 `app:`과 같은 특정 범위를 대상으로 하거나, 에이전트의 최종 텍스트 응답과 연결되지 않은 업데이트를 해야 할 때는 상태 변경 딕셔너리(state_delta)를 수동으로 구성하고 추가하는 Event의 EventActions에 포함시켜야 합니다. 한 가지 예를 살펴보겠습니다.

````python
import time
from google.adk.tools.tool_context import ToolContext
from google.adk.sessions import InMemorySessionService

# --- Define the Recommended Tool-Based Approach ---
def log_user_login(tool_context: ToolContext) -> dict:
   """
   Updates the session state upon a user login event.
   This tool encapsulates all state changes related to a user
login.
   Args:
       tool_context: Automatically provided by ADK, gives access
to session state.
   Returns:
       A dictionary confirming the action was successful.
   """
   # Access the state directly through the provided context.
   state = tool_context.state

   # Get current values or defaults, then update the state.
   # This is much cleaner and co-locates the logic.
   login_count = state.get("user:login_count", 0) + 1
   state["user:login_count"] = login_count
   state["task_status"] = "active"
   state["user:last_login_ts"] = time.time()
   state["temp:validation_needed"] = True

   print("State updated from within the `log_user_login` tool.")
````

이 코드는 애플리케이션에서 사용자 세션 상태를 관리하기 위한 도구 기반 접근법을 보여줍니다. `log_user_login` 함수를 도구로 정의하며, 이 도구는 사용자가 로그인할 때 세션 상태를 업데이트하는 역할을 합니다. 함수는 ADK가 제공하는 ToolContext 객체를 받아 세션의 상태 딕셔너리에 접근하고 수정합니다. 도구 내부에서 `user:login_count`를 증가시키고, `task_status`를 "active"로 설정하며, `user:last_login_ts`(타임스탬프)를 기록하고, 임시 플래그 `temp:validation_needed`를 추가합니다.

코드의 데모 부분은 이 도구가 어떻게 사용될지를 시뮬레이션합니다. 인메모리 세션 서비스를 설정하고 미리 정의된 상태로 초기 세션을 생성합니다. ADK Runner가 도구를 실행할 환경을 모방하기 위해 ToolContext를 수동으로 생성합니다. `log_user_login` 함수가 이 모의 컨텍스트와 함께 호출됩니다. 마지막으로, 도구 실행에 의해 상태가 업데이트되었음을 보여주기 위해 세션을 다시 검색합니다. 목표는 도구 내에서 상태 변경을 캡슐화하는 것이 도구 외부에서 상태를 직접 조작하는 것보다 코드를 더 깔끔하고 구조화되게 만든다는 것을 보여주는 것입니다.

세션을 검색한 후 `session.state` 딕셔너리를 직접 수정하는 것은 표준 이벤트 처리 메커니즘을 우회하므로 강력히 권장하지 않습니다. 이러한 직접 변경은 세션의 이벤트 이력에 기록되지 않고, 선택된 SessionService에 의해 영속화되지 않을 수 있으며, 동시성 문제를 일으킬 수 있고, 타임스탬프와 같은 필수 메타데이터를 업데이트하지 않습니다. 세션 상태를 업데이트하는 권장 방법은 LlmAgent의 `output_key` 파라미터를 사용하거나(에이전트의 최종 텍스트 응답에 한정), `session_service.append_event()`를 통해 이벤트를 추가할 때 `EventActions.state_delta`에 상태 변경을 포함하는 것입니다. `session.state`는 주로 기존 데이터를 읽는 용도로 사용해야 합니다.

정리하면, 상태를 설계할 때는 단순하게 유지하고, 기본 데이터 유형을 사용하며, 키에 명확한 이름을 부여하고 접두사를 올바르게 사용하고, 깊은 중첩을 피하며, 항상 `append_event` 프로세스를 사용하여 상태를 업데이트해야 합니다.

## 메모리: MemoryService를 통한 장기 지식

에이전트 시스템에서 세션(Session) 구성요소는 단일 대화의 현재 채팅 이력(이벤트)과 임시 데이터(상태)를 유지합니다. 그러나 에이전트가 여러 상호작용에 걸쳐 정보를 보유하거나 외부 데이터에 접근하려면 장기 지식 관리가 필요합니다. 이를 MemoryService가 담당합니다.

````python
# Example: Using InMemoryMemoryService
# This is suitable for local development and testing where data
# persistence across application restarts is not required.
# Memory content is lost when the app stops.
from google.adk.memory import InMemoryMemoryService
memory_service = InMemoryMemoryService()
````

세션(Session)과 상태(State)는 단일 채팅 세션의 단기 기억으로 개념화될 수 있는 반면, MemoryService가 관리하는 장기 지식은 지속적이고 검색 가능한 저장소로 기능합니다. 이 저장소는 여러 과거 상호작용이나 외부 소스의 정보를 포함할 수 있습니다. BaseMemoryService 인터페이스로 정의된 MemoryService는 이 검색 가능한 장기 지식을 관리하기 위한 표준을 확립합니다. 주요 기능에는 정보 추가(세션에서 콘텐츠를 추출하여 `add_session_to_memory` 메서드로 저장)와 정보 검색(에이전트가 저장소를 조회하고 `search_memory` 메서드로 관련 데이터를 수신)이 포함됩니다.

ADK는 이 장기 지식 저장소를 만들기 위한 여러 구현을 제공합니다. InMemoryMemoryService는 테스트 목적에 적합한 임시 저장소 솔루션을 제공하지만, 데이터는 애플리케이션 재시작 시 보존되지 않습니다. 프로덕션 환경에서는 일반적으로 VertexAiRagMemoryService가 사용됩니다. 이 서비스는 Google Cloud의 검색 증강 생성(RAG) 서비스를 활용하여 확장 가능하고 영속적이며 시맨틱 검색 기능을 지원합니다(RAG에 대해서는 14장을 참고하십시오).

````text
# Example: Using VertexAiRagMemoryService
# This is suitable for scalable production on GCP, leveraging
# Vertex AI RAG (Retrieval Augmented Generation) for persistent,
# searchable memory.
# Requires: pip install google-adk[vertexai], GCP
# setup/authentication, and a Vertex AI RAG Corpus.
from google.adk.memory import VertexAiRagMemoryService

# The resource name of your Vertex AI RAG Corpus
RAG_CORPUS_RESOURCE_NAME =
"projects/your-gcp-project-id/locations/us-central1/ragCorpora/your-c
orpus-id" # Replace with your Corpus resource name
````

````python
# Optional configuration for retrieval behavior
SIMILARITY_TOP_K = 5 # Number of top results to retrieve
VECTOR_DISTANCE_THRESHOLD = 0.7 # Threshold for vector similarity

memory_service = VertexAiRagMemoryService(
   rag_corpus=RAG_CORPUS_RESOURCE_NAME,
   similarity_top_k=SIMILARITY_TOP_K,
   vector_distance_threshold=VECTOR_DISTANCE_THRESHOLD
)
# When using this service, methods like add_session_to_memory
# and search_memory will interact with the specified Vertex AI
# RAG Corpus.
````

## 실습 코드: LangChain 및 LangGraph에서의 메모리 관리

LangChain과 LangGraph에서 메모리는 지능적이고 자연스러운 대화형 애플리케이션을 만드는 데 중요한 구성요소입니다. 이를 통해 AI 에이전트는 과거 상호작용에서 정보를 기억하고, 피드백에서 학습하며, 사용자 선호도에 적응할 수 있습니다. LangChain의 메모리 기능은 저장된 이력을 참조하여 현재 프롬프트를 풍부하게 하고 최신 교환을 미래 사용을 위해 기록함으로써 이를 위한 기반을 제공합니다. 에이전트가 더 복잡한 작업을 처리할수록, 이 기능은 효율성과 사용자 만족도 모두에 필수적이 됩니다.

**단기 기억:** 이는 스레드 범위로, 단일 세션이나 스레드 내에서 진행 중인 대화를 추적합니다. 즉각적인 컨텍스트를 제공하지만, 전체 이력은 LLM의 컨텍스트 윈도우에 부담을 줄 수 있어 오류나 낮은 성능으로 이어질 수 있습니다. LangGraph는 에이전트 상태의 일부로 단기 기억을 관리하며, 체크포인터를 통해 영속화되어 스레드를 언제든지 재개할 수 있습니다.

**장기 기억:** 이는 세션에 걸쳐 사용자 특화 또는 애플리케이션 수준의 데이터를 저장하며, 대화 스레드 간에 공유됩니다. 커스텀 "네임스페이스"에 저장되어 어떤 스레드에서든 언제든지 회상할 수 있습니다. LangGraph는 장기 기억을 저장하고 회상하기 위한 저장소를 제공하여 에이전트가 지식을 무기한 보유할 수 있게 합니다.

LangChain은 수동 제어부터 체인 내의 자동화된 통합에 이르기까지 대화 이력을 관리하기 위한 여러 도구를 제공합니다.

### ChatMessageHistory: 수동 메모리 관리

공식적인 체인 외부에서 대화 이력을 직접적이고 간단하게 제어하려면 ChatMessageHistory 클래스가 이상적입니다. 이를 통해 대화 교환을 수동으로 추적할 수 있습니다.

````python
from langchain.memory import ChatMessageHistory

# Initialize the history object
history = ChatMessageHistory()

# Add user and AI messages
history.add_user_message("I'm heading to New York next week.")
history.add_ai_message("Great! It's a fantastic city.")

# Access the list of messages
print(history.messages)
````

### ConversationBufferMemory: 체인을 위한 자동 메모리

메모리를 체인에 직접 통합하려면 ConversationBufferMemory가 일반적인 선택입니다. 대화의 버퍼를 유지하고 프롬프트에서 사용 가능하게 합니다. 두 가지 주요 파라미터로 동작을 커스터마이징할 수 있습니다.

- `memory_key`: 채팅 이력을 보유할 프롬프트의 변수 이름을 지정하는 문자열입니다. 기본값은 "history"입니다.
- `return_messages`: 이력의 형식을 지정하는 불리언입니다. `False`(기본값)이면 표준 LLM에 적합한 단일 형식 문자열을 반환합니다. `True`이면 채팅 모델에 권장되는 메시지 객체 목록을 반환합니다.

````python
from langchain.memory import ConversationBufferMemory

# Initialize memory
memory = ConversationBufferMemory()

# Save a conversation turn
memory.save_context({"input": "What's the weather like?"}, {"output":
"It's sunny today."})

# Load the memory as a string
print(memory.load_memory_variables({}))
````

이 메모리를 LLMChain에 통합하면 모델이 대화 이력에 접근하여 맥락적으로 관련된 응답을 제공할 수 있습니다.

````python
from langchain_openai import OpenAI
from langchain.chains import LLMChain
from langchain.prompts import PromptTemplate
from langchain.memory import ConversationBufferMemory

# 1. Define LLM and Prompt
llm = OpenAI(temperature=0)
template = """You are a helpful travel agent.

Previous conversation:
{history}

New question: {question}
Response:"""
prompt = PromptTemplate.from_template(template)

# 2. Configure Memory
# The memory_key "history" matches the variable in the prompt
````

## 장기 기억의 유형

장기 기억은 시스템이 서로 다른 대화에 걸쳐 정보를 보유할 수 있게 하여 더 깊은 수준의 컨텍스트와 개인화를 제공합니다. 인간 기억과 유사한 세 가지 유형으로 나눌 수 있습니다.

- **의미 기억(Semantic Memory): 사실 기억:** 이는 사용자 선호도나 도메인 지식과 같은 특정 사실과 개념을 보유하는 것을 포함합니다. 에이전트의 응답을 근거 있게 만들어 더 개인화되고 관련성 높은 상호작용으로 이어집니다. 이 정보는 지속적으로 업데이트되는 사용자 "프로필"(JSON 문서)이나 개별 사실 문서의 "컬렉션"으로 관리될 수 있습니다.

- **에피소드 기억(Episodic Memory): 경험 기억:** 이는 과거 사건이나 행동을 회상하는 것을 포함합니다. AI 에이전트에서 에피소드 기억은 종종 작업 수행 방법을 기억하는 데 사용됩니다. 실제로는 퓨샷 예제 프롬프팅을 통해 자주 구현되며, 에이전트가 과거의 성공적인 상호작용 시퀀스에서 학습하여 작업을 올바르게 수행합니다.

- **절차 기억(Procedural Memory): 규칙 기억:** 이는 작업 수행 방법에 대한 기억으로, 에이전트의 핵심 지시사항과 행동이며 종종 시스템 프롬프트에 포함됩니다. 에이전트가 자체 프롬프트를 수정하여 적응하고 개선하는 것이 일반적입니다. 효과적인 기법은 "반성(Reflection)"으로, 에이전트에게 현재 지시사항과 최근 상호작용을 제시한 후 자체 지시사항을 개선하도록 요청합니다.

아래는 LangGraph BaseStore에 저장된 절차 기억을 업데이트하기 위해 에이전트가 반성을 사용하는 방법을 보여주는 의사 코드입니다.

## Vertex Memory Bank

Vertex AI Agent Engine의 관리형 서비스인 Memory Bank는 에이전트에게 영속적인 장기 기억을 제공합니다. 이 서비스는 Gemini 모델을 사용하여 대화 이력을 비동기적으로 분석하여 핵심 사실과 사용자 선호도를 추출합니다.

이 정보는 영속적으로 저장되고, 사용자 ID와 같이 정의된 범위로 구성되며, 새 데이터를 통합하고 모순을 해결하기 위해 지능적으로 업데이트됩니다. 새 세션을 시작할 때, 에이전트는 전체 데이터 회상이나 임베딩을 사용한 유사도 검색을 통해 관련 기억을 검색합니다. 이 프로세스를 통해 에이전트는 세션 간 연속성을 유지하고 회상된 정보를 기반으로 응답을 개인화할 수 있습니다.

에이전트의 Runner는 먼저 초기화되는 VertexAiMemoryBankService와 상호작용합니다. 이 서비스는 에이전트의 대화 중에 생성된 기억의 자동 저장을 처리합니다. 각 기억은 고유한 USER_ID와 APP_NAME으로 태그가 지정되어 향후 정확한 검색을 보장합니다.

````text
from google.adk.memory import VertexAiMemoryBankService

agent_engine_id = agent_engine.api_resource.name.split("/")[-1]

memory_service = VertexAiMemoryBankService(
   project="PROJECT_ID",
   location="LOCATION",
   agent_engine_id=agent_engine_id
)

session = await session_service.get_session(
   app_name=app_name,
   user_id="USER_ID",
   session_id=session.id
)
await memory_service.add_session_to_memory(session)
````

Memory Bank는 Google ADK와의 원활한 통합을 제공하여 즉시 사용 가능한 경험을 제공합니다. LangGraph 및 CrewAI와 같은 다른 에이전트 프레임워크 사용자를 위해서도 Memory Bank는 직접 API 호출을 통한 지원을 제공합니다. 이러한 통합을 보여주는 온라인 코드 예제를 관심 있는 독자들이 쉽게 이용할 수 있습니다.

## 한눈에 보기

**무엇을:** 에이전트 시스템은 복잡한 작업을 수행하고 일관된 경험을 제공하기 위해 과거 상호작용에서 정보를 기억해야 합니다. 메모리 메커니즘이 없으면 에이전트는 상태가 없어 대화 컨텍스트를 유지하거나, 경험에서 학습하거나, 사용자를 위해 응답을 개인화할 수 없습니다. 이는 에이전트를 근본적으로 단순하고 일회성 상호작용으로 제한하여 다단계 프로세스나 진화하는 사용자 요구를 처리하지 못하게 합니다. 핵심 문제는 단일 대화의 즉각적이고 임시적인 정보와 시간이 지남에 따라 수집된 광대하고 영속적인 지식을 모두 효과적으로 관리하는 방법입니다.

**왜:** 표준화된 해결책은 단기 저장소와 장기 저장소를 구분하는 이중 구성요소 메모리 시스템을 구현하는 것입니다. 단기 컨텍스트 기억은 대화 흐름을 유지하기 위해 LLM의 컨텍스트 윈도우 내에 최근 상호작용 데이터를 보유합니다. 영속화해야 하는 정보의 경우, 장기 기억 솔루션은 효율적인 시맨틱 검색을 위해 외부 데이터베이스, 종종 벡터 저장소를 사용합니다. Google ADK와 같은 에이전틱 프레임워크는 이를 관리하기 위한 특정 구성요소를 제공합니다. 예를 들어, 대화 스레드를 위한 세션(Session)과 임시 데이터를 위한 상태(State)가 있습니다. 전용 MemoryService는 장기 지식 기반과 인터페이스하는 데 사용되어, 에이전트가 현재 컨텍스트에 관련된 과거 정보를 검색하고 통합할 수 있게 합니다.

**경험칙:** 에이전트가 단일 질문에 답하는 것 이상을 해야 할 때 이 패턴을 사용하십시오. 대화 전반에 걸쳐 컨텍스트를 유지하거나, 다단계 작업에서 진행 상황을 추적하거나, 사용자 선호도와 이력을 회상하여 상호작용을 개인화해야 하는 에이전트에게 필수적입니다. 에이전트가 과거 성공, 실패, 또는 새로 획득한 정보를 기반으로 학습하거나 적응할 것으로 예상될 때마다 메모리 관리를 구현하십시오.

## 핵심 정리

메모리 관리의 주요 사항을 빠르게 정리합니다.

- 메모리는 에이전트가 사물을 추적하고, 학습하며, 상호작용을 개인화하는 데 매우 중요합니다.
- 대화형 AI는 단일 채팅 내의 즉각적인 컨텍스트를 위한 단기 기억과 여러 세션에 걸친 영속적 지식을 위한 장기 기억 모두에 의존합니다.
- 단기 기억(즉각적인 것)은 일시적이며, 종종 LLM의 컨텍스트 윈도우나 프레임워크가 컨텍스트를 전달하는 방식에 의해 제한됩니다.
- 장기 기억(계속 남는 것)은 벡터 데이터베이스와 같은 외부 저장소를 사용하여 서로 다른 채팅에 걸쳐 정보를 저장하며, 검색을 통해 접근됩니다.
- ADK와 같은 프레임워크는 메모리를 관리하기 위해 세션(Session)(채팅 스레드), 상태(State)(임시 채팅 데이터), MemoryService(검색 가능한 장기 지식)와 같은 특정 부분을 가지고 있습니다.
- ADK의 SessionService는 채팅 세션의 전체 생명주기를 처리하며, 이력(이벤트)과 임시 데이터(상태)를 포함합니다.
- ADK의 `session.state`는 임시 채팅 데이터를 위한 딕셔너리입니다. 접두사(`user:`, `app:`, `temp:`)는 데이터가 어디에 속하는지와 영속 여부를 알려줍니다.
- ADK에서는 상태 딕셔너리를 직접 변경하지 않고, 이벤트를 추가할 때 `EventActions.state_delta` 또는 `output_key`를 사용하여 상태를 업데이트해야 합니다.
- ADK의 MemoryService는 정보를 장기 저장소에 저장하고 에이전트가 종종 도구를 사용하여 검색할 수 있도록 합니다.
- LangChain은 에이전트가 즉각적인 컨텍스트를 회상할 수 있도록 단일 대화의 이력을 프롬프트에 자동으로 주입하기 위해 ConversationBufferMemory와 같은 실용적인 도구를 제공합니다.
- LangGraph는 저장소를 사용하여 서로 다른 사용자 세션에 걸쳐 의미론적 사실, 에피소드 경험, 또는 업데이트 가능한 절차적 규칙을 저장하고 검색함으로써 고급 장기 기억을 가능하게 합니다.
- Memory Bank는 대화 이력을 자동으로 추출·저장·회상하여 Google ADK, LangGraph, CrewAI와 같은 프레임워크에서 개인화되고 지속적인 대화를 가능하게 하는 에이전트에게 영속적인 장기 기억을 제공하는 관리형 서비스입니다.

## 결론

이 장에서는 에이전트 시스템의 메모리 관리라는 중요한 역할을 심층적으로 다루며, 일시적인 컨텍스트와 장기간 지속되는 지식 간의 차이를 명확히 설명하였습니다. 이러한 메모리 유형이 어떻게 구성되고 더 스마트한 에이전트를 구축하는 데 어디서 활용되는지를 살펴보았습니다. Google ADK가 세션(Session), 상태(State), MemoryService와 같은 특정 구성요소를 통해 이를 어떻게 처리하는지 자세히 살펴보았습니다. 이제 에이전트가 단기 및 장기 모두에서 어떻게 기억하는지를 다루었으므로, 에이전트가 어떻게 학습하고 적응하는지로 넘어갈 수 있습니다. 다음 패턴 "학습 및 적응"은 에이전트가 새로운 경험이나 데이터를 기반으로 사고 방식, 행동 방식, 또는 알고 있는 것을 어떻게 변화시키는지에 관한 것입니다.
